Security Bulletin

Бюллетень по безопасности (Майкрософт) MS10-049 - Критическое
=============================================================

Уязвимости в пакете безопасности SChannel делают возможным удаленное выполнение кода (980436)
---------------------------------------------------------------------------------------------

Дата публикации: 10 августа 2010 г. | Дата обновления: 1 сентября 2010 г.

**Версия:** 1.1

### Общие сведения

#### Аннотация

Это обновление для системы безопасности устраняет одну опубликованную и одну обнаруженную пользователями уязвимость в пакете безопасности Secure Channel (SChannel) в ОС Windows. Наиболее опасная из этих уязвимостей позволяет злоумышленнику удаленно выполнять код в том случае, если пользователь посетит через веб-браузер веб-сайт, специально созданный для того чтобы воспользоваться этой уязвимостью. Однако в любом случае злоумышленник не может заставить пользователей посетить эти веб-сайты. Злоумышленник может создать специальный веб-сайт для использования данной уязвимости через браузер Internet Explorer, а затем убедить пользователя посетить его.

Это обновление для системы безопасности имеет уровень серьезности "критический" для всех поддерживаемых выпусков Windows XP и Windows Server 2003 и уровень серьезности "существенный" для всех поддерживаемых выпусков Windows Vista, Windows Server2008, Windows 7 и Windows Server 2008 R2. Дополнительные сведения см. в подразделе **Подвержены и не подвержены уязвимости** этого раздела.

Это обновление для системы безопасности устраняет уязвимость, реализуя требования документа RFC 5746 и устанавливая дополнительную проверку SSL-откликов, возвращаемых сервером. Дополнительные сведения об этих уязвимостях см. в подразделе ответов на часто задаваемые вопросы для каждой уязвимости, упомянутой в разделе **Сведения об уязвимостях**.

Кроме того, это обновление устраняет уязвимость, впервые описанную в [выпуске 977377 советов Майкрософт по безопасности](http://technet.microsoft.com/security/advisory/977377).

**Рекомендация**. У большинства клиентов включено автоматическое обновление, поэтому нет необходимости предпринимать какие-либо действия: данное обновление для системы безопасности загрузится и установится автоматически. Клиентам, у которых не включено автоматическое обновление, необходимо проверить наличие обновлений и установить это обновление вручную. Дополнительные сведения об особых параметрах конфигурации автоматического обновления см. в [статье 294871 базы знаний Майкрософт](http://support.microsoft.com/kb/294871).

При установке на предприятиях, а также администраторам или конечным пользователям, которые хотят применить данное обновление для системы безопасности вручную, корпорация Майкрософт рекомендует сделать это немедленно, используя программное обеспечение для управления обновлениями или проверив его наличие в [Центре обновления Майкрософт](http://go.microsoft.com/fwlink/?linkid=40747).

См. также раздел **Руководство и средства по диагностике и развертыванию** этого бюллетеня.

**Известные проблемы**. Нет

Часто задаваемые вопросы о данном обновлении безопасности
---------------------------------------------------------

<span></span>
**Где найти дополнительные сведения о файлах?**  
Местонахождение дополнительных сведений о файлах см. в справочных таблицах раздела **Развертывание обновления для системы безопасности**.

**Почему это обновление устраняет несколько уязвимостей системы безопасности, о которых уже сообщалось?**  
Это обновление разработано для нескольких уязвимостей, поскольку для устранения их причин требуется выполнить изменения в связанных файлах. Вместо установки нескольких почти одинаковых обновлений пользователям необходимо установить только данное обновление.

**На моем компьютере установлена более ранняя версия программного обеспечения, описанного в этом бюллетене по безопасности. Что мне нужно сделать?**  
Продукты, перечисленные в этом бюллетене, проверены на наличие уязвимости в тех или иных версиях. Жизненные циклы поддержки прочих версий программного обеспечения истекли. Дополнительные сведения о жизненном цикле продуктов см. на веб-странице [сроков поддержки продуктов Майкрософт](http://go.microsoft.com/fwlink/?linkid=21742).

Пользователям более ранних версий программного обеспечения следует срочно перейти на поддерживаемые версии, чтобы снизить вероятность наличия в системе уязвимостей. Чтобы определить стадии жизненного цикла поддержки для вашего выпуска программного обеспечения, перейдите на веб-страницу [Выберите продукт для отображения информации о стадиях жизненного цикла](http://go.microsoft.com/fwlink/?linkid=169555). Дополнительные сведения о пакетах обновления для этих выпусков программного обеспечения см. на веб-странице [Пакеты обновления, поддерживаемые на стадиях жизненного цикла](http://go.microsoft.com/fwlink/?linkid=89213).

Для получения сведений о возможных вариантах обслуживания более ранних версий программного обеспечения свяжитесь с представителем группы по работе с заказчиками корпорации Майкрософт, ее техническим менеджером или представителем соответствующей партнерской компании Майкрософт. Пользователи, у которых нет договора типа Alliance, Premier или Authorized, могут обратиться в местное представительство корпорации Майкрософт. Для получения контактной информации посетите веб-сайт [Microsoft Worldwide Information](http://go.microsoft.com/fwlink/?linkid=33329), выберите страну в списке "Контактная информация" и нажмите кнопку **Go** (Перейти), чтобы просмотреть список телефонных номеров. Дозвонившись, попросите связать вас с менеджером по продажам службы поддержки Premier. Дополнительные сведения см. в разделе [Часто задаваемые вопросы о политике поддержки продуктов Майкрософт на разных стадиях жизненного цикла](http://go.microsoft.com/fwlink/?linkid=169557).

Уязвимость, связанная с повторным согласованием TLS/SSL (CVE-2009-3555)  
-----------------------------------------------------------------------
  
<span></span>
В протоколе TLS/SSL, реализованном в компоненте проверки подлинности Microsoft Windows SChannel, существует уязвимость, делающая возможным спуфинг. Злоумышленник, воспользовавшейся этой уязвимостью, может внедрять информацию в защищенное подключение по протоколу TLS/SSL, посылая трафик для спуфинга клиента, прошедшего проверку подлинности. Данное обновление устраняет уязвимость, ранее описанную в [выпуске 977377 советов Майкрософт по безопасности](http://technet.microsoft.com/security/advisory/977377).
  
В списке "Common Vulnerabilities and Exposures" данной уязвимости присвоен номер [CVE-2009-3555](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2009-3555).
  
#### Смягчающие факторы для уязвимости, связанной с повторным согласованием TLS/SSL (CVE-2009-3555)
  
К факторам, снижающим опасность уязвимости, относятся параметр, стандартная конфигурация или общие рекомендации, которые при использовании по умолчанию могут снизить опасность использования уязвимости. Следующие смягчающие факторы могут снизить опасность использования уязвимости.
  
-   Веб-сайты, в которых не содержится контента для передачи по SSL, и которые передают данные только по протоколу HTTP (незашифрованный текс), не подвержены данной уязвимости.  
-   Службы IIS 6 и IIS 7 не допускают повторного согласования по инициативе клиента. Это значительно сокращает возможности для атаки. Вероятность уязвимости SSL-сайтов, на которых пользователь вводит имя пользователя и пароль, очень мала. Наиболее вероятная ситуация, при которой возможен спуфинг при работе служб IIS 6 и IIS 7, заключается в использовании взаимной проверки подлинности на основе клиентских сертификатов. Такая ситуация встречается довольно редко.  
-   Пользователи уязвимы, только когда злоумышленник способен на основе другой уязвимости успешно осуществить атаку "злоумышленник в середине", например атаку на локальную подсеть или DNS-спуфинг.
  
#### Временные решения для уязвимости, связанной с повторным согласованием TLS/SSL (CVE-2009-3555)
  
К временным решениям относятся изменения параметров или конфигурации, не приводящие к полному устранению уязвимости, но позволяющие блокировать известные направления атак до того, как будет установлено обновление. Корпорация Майкрософт проверила представленные ниже временные решения и сообщила, снижают ли они функциональные возможности.
  
-   **Включение SSLAlwaysNegoClientCert в IIS 6 и более поздних версиях**
  
    Безопасность веб-серверов с IIS 6 или более поздней версией, уязвимых из-за требования взаимной проверки подлинности путем запроса сертификата клиента, может быть укреплена путем установки флага SSLAlwaysNegoClientCert. В результате при исходном подключении служба IIS запрашивает у клиента сертификат и не требует повторного согласования, инициированного сервером.
  
    **Побочные эффекты:** После установки этого флажка клиент должен проходить проверку подлинности перед загрузкой любого элемента из веб-сайта, защищенного протоколом SSL. В результате при подключении к веб-сайту, защищенному протоколом SSL, браузер всегда запрашивает у пользователя сертификат клиента. Если взаимная проверка подлинности *не* требуется для защищенного протоколом SSL содержимого, пользователи, которым предложено выбрать сертификат, могут просто отклонить данное диалоговое окно, чтобы просматривать данное содержимое по SSL-соединению, не выбирая сертификат клиента для проверки подлинности.
  
    **Для IIS 6:**
  
    Откройте окно командной строки с расширенными правами администратора и в папке "c:\\inetpub\\adminscripts" выполните следующую команду:
  
    `adsutil.vbs SET w3svc/<N>/SSLAlwaysNegoClientCert true`
  
    Здесь &lt;N&gt; обозначает число настраиваемых веб-сайтов (т.е. "Веб-сайт по умолчанию" — 1, следующий веб-сайт — 2 и т.д.). Например, чтобы защитить "Веб-сайт по умолчанию", созданный службой IIS, используется следующая команда:
  
    `adsutil.vbs SET w3svc/1/SSLAlwaysNegoClientCert true`
  
    **Для IIS 7:**
  
    Сохраните в файл "Enable\_SSL\_Renegotiate\_Workaround.js" приведенный ниже текст.
  
    `var vdirObj=GetObject("IIS://localhost/W3svc/1");// вместо цифры 1 в этой строке укажите номер настраиваемого веб-сайтаWScript.Echo("Value of SSLAlwaysNegoClientCert Before: " + vdirObj.SSLAlwaysNegoClientCert);vdirObj.Put("SSLAlwaysNegoClientCert", true);vdirObj.SetInfo();WScript.Echo("Value of SSLAlwaysNegoClientCert After: " + vdirObj.SSLAlwaysNegoClientCert);`
  
    Войдите в систему с расширенными правами администратора и в командной строке выполните следующую команду:
  
    `cscript.exe enable_ssl_renegotiate_workaround.js`
  
#### Часто задаваемые вопросы об уязвимости, связанной с повторным согласованием TLS/SSL (CVE-2009-3555)
  
**Какова область воздействия этой уязвимости?**    
Это уязвимость, которая делает возможным спуфинг. Злоумышленник, воспользовавшейся этой уязвимостью, может внедрять информацию в защищенное подключение по протоколу TLS/SSL, посылая трафик для спуфинга клиента, прошедшего проверку подлинности.
  
**В чем причина уязвимости?**    
Протокол TLS (см. RFC 2246) описывает функцию повторного согласования, позволяющую любому из одноранговых узлов в любой момент повторно согласовать параметры защищенного соединения. Злоумышленник, способный осуществить другую атаку, такую как DNS-спуфинг или атака на локальную подсеть, чтобы стать "злоумышленником в середине" на таком соединении, может воспользоваться функцией повторного согласования с целью добавления команд (зависящих от приложения) в начало настраиваемого действительного сеанса TLS.
  
В особых случаях сервер может выполнить такие команды в контексте прошедшего проверку TLS/SSL клиента. В Windows протоколы TLS и SSL реализуются пакетом безопасности Secure Channel (SChannel).
  
**Почему стандартная установка служб IIS 6.0 и более поздней версии не подвержена данной уязвимости?**  
Использование этой уязвимости возможно, только если один из одноранговых узлов разрешает повторное согласование параметров TLS/SSL. IIS предотвращает любые повторные согласования TLS/SSL, инициированные клиентом. Кроме того, IIS инициирует попытку повторного согласования на стороне сервера только когда данная служба настроена на использование взаимной проверки подлинности на основе сертификатов, что не является стандартной настройкой.
  
Обратите внимание, что в развертываниях, в которых действительное TLS/SSL-соединение завершается на сервере IIS напрямую, а на стороннем SSL-терминаторе или обратном прокси, соединение может оставаться восприимчивым к данной уязвимости.
  
**Что такое атака типа "злоумышленник в середине"?**  
Во время атаки типа "злоумышленник в середине" злоумышленник перенаправляет трафик между двумя пользователями через свой компьютер, не имея информации об этих двух взаимодействующих пользователях. Все участвующие в таком взаимодействии пользователи невольно обмениваются трафиком со злоумышленником, думая, что взаимодействуют только с предполагаемым пользователем. При использовании данной уязвимости "злоумышленник в середине" не может читать, расшифровывать или изменять зашифрованные данные, передаваемые между клиентом и сервером. Он может лишь ввести в сеанс TLS запрос, выполняемый в контексте клиента.
  
**Что такое взаимная проверка подлинности?**  
Взаимная проверка подлинности — это функция безопасности, требующая, чтобы клиент и сервер подтвердили свою подлинность друг перед другом, прежде чем передавать данные приложения по соединению между клиентом и сервером.
  
Взаимная проверка подлинности требует, чтобы клиент и сервер подтвердили свою подлинность друг перед другом, прежде чем будут использоваться любые функции приложения. Подлинность может быть подтверждена криптографическими средствами, например с помощью инфраструктура открытого ключа и цифровых сертификатов.
  
**Что такое TLS?**  
Протокол подтверждения соединения TLS отвечает за проверку подлинности и обмен ключами, необходимые для установления или возобновления безопасных соединений. При установлении безопасного соединения протокол подтверждения соединения обеспечивает управление следующими процессами:
  
согласование комплекта шифров;
  
проверка подлинности сервера и (необязательно) клиента;
  
обмен сведениями о ключах сеанса.
  
Дополнительные сведения см. в статье [Как работает TLS/SSL](http://technet.microsoft.com/en-us/library/cc783349(ws.10).aspx) на веб-сайте Technet.
  
**Что такое SSL?**  
SSL — предшественник протокола Transport Layer Security. Он выполняет те же функции и поддерживает защищенные сетевые взаимодействия, используя технологию создания открытого и секретного ключей.
  
Дополнительные сведения см. в статье [Как работает TLS/SSL](http://technet.microsoft.com/en-us/library/cc783349(ws.10).aspx) на веб-сайте Technet.
  
**Что такое повторное согласование TLS/SSL?**  
Повторное согласование — это функция протоколов TLS и SSL, позволяющая любому из одноранговых узлов повторно согласовать параметры криптографического соединения, например ключи шифрования.
  
**Что такое пакет безопасности SChannel?**    
Пакет безопасности безопасного канала Schannel — это поставщик поддержки безопасности (SSP), реализующий стандартные интернет-протоколы проверки подлинности SSL и TLS. Эти компоненты используются для безопасного обмена данными при поддержке некоторых общих интернет-приложений и сетевых приложений, например веб-браузеров. Безопасный канал Schannel является частью пакета безопасности, предоставляющего службу проверки подлинности для обеспечения безопасного обмена данными между клиентом и сервером.
  
Дополнительные сведения см. в статье MSDN [Secure Channel](http://msdn.microsoft.com/en-us/library/aa380123(vs.85).aspx) (Безопасный канал).
  
**Что может сделать злоумышленник, который воспользуется этой уязвимостью?**    
В уязвимых развертываниях злоумышленник может воспользоваться данной уязвимостью, чтобы добавить некоторое содержимое в начало легального запроса к серверу, инициированного клиентом, в контексте действительного сеанса с проверкой подлинности по протоколу TLS/SSL. Данная уязвимость не позволяет злоумышленнику читать, расшифровывать или изменять шифрованный трафик между клиентом и сервером.
  
**Каким образом злоумышленник может воспользоваться этой уязвимостью?**    
Злоумышленник может воспользоваться данной уязвимостью, организовав атаку типа "человек в середине" между клиентом и сервером, а затем, при попытке повторного согласования SSL, прерывая подключение между ними. В этот момент времени злоумышленник может внедрить в защищенный канал ограниченный набор данных, который будет интерпретирован сервером как поступивший от доверенного клиента.
  
**Какие системы в первую очередь подвержены риску?**    
Данной уязвимости прежде всего подвержены клиенты, использующиеся в веб-браузерах, а также соответствующие веб-серверы; при этом в некоторых вариациях атакам могут быть подвержены другие протоколы с защитой TLS/SSL.
  
**Как действует обновление?**    
Данное обновление устраняет уязвимость, реализуя требования документа RFC 5746. В этом документе представлено расширение TLS, которое криптографическим образом привязывает повторные согласования TLS к тому подключению TLS, по которому они изначально выполняются. Это позволяет проводить проверку подлинности, а затем проверять ее, а также предотвращает атаку.
  
**Была ли эта уязвимость опубликована до выпуска этого бюллетеня безопасности?**    
Да. О данной уязвимости сообщалось в открытых источниках. В списке "Common Vulnerabilities and Exposures" данной уязвимости присвоен номер [CVE-2009-3555](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2009-3555).
  
**Получала ли корпорация Майкрософт к моменту выпуска этого бюллетеня по безопасности какие-либо сведения о том, что уязвимость была использована злоумышленниками?**    
Нет. Корпорация Майкрософт не получала никакой информации, указывающей на то, что до выпуска этого бюллетеня по безопасности данная уязвимость была открыто использована для организации атак на пользовательские компьютеры.

**Что такое SSL?**  
SSL — предшественник протокола Transport Layer Security. Он выполняет те же функции и поддерживает защищенные сетевые взаимодействия, используя технологию создания открытого и секретного ключей.
  
Дополнительные сведения см. в статье [Как работает TLS/SSL](http://technet.microsoft.com/en-us/library/cc783349(ws.10).aspx) на веб-сайте Technet.
  
**Что может сделать злоумышленник, который воспользуется этой уязвимостью?**    
Воспользовавшись уязвимостью, злоумышленник теоретически может получить полный контроль над системой. Однако попытки воспользоваться ею, вероятнее всего, приведут к отказу в обслуживании. Система не будет соединяться с веб-узлами и другими сетевыми ресурсами по протоколу SSL или TLS, пока не будет перезагружена.
  
**Каким образом злоумышленник может воспользоваться этой уязвимостью?**    
Злоумышленник может воспользоваться этой уязвимостью через браузер, создав специальный веб-узел и убедив пользователя посетить его. Злоумышленник не может заставить пользователей посетить эти веб-сайты. Он старается склонить пользователей к этому, обычно убеждая их щелкнуть ссылку, ведущую на его веб-сайт, в сообщении электронной почты или в запросе службы мгновенных сообщений.
  
**Какие системы в первую очередь подвержены риску?**    
Вредоносное действие возможно только при посещении веб-сайтов пользователями, выполнившими вход в систему. Поэтому все системы, в которых часто используются обозреватель или приложения, работающие с протоколами SSL и TLS (например, рабочие станции или серверы терминалов), подвержены наибольшему риску воздействия этой уязвимости.
  
**Как действует обновление?**    
Данное обновление реализует дополнительную проверку SSL-откликов, возвращаемых сервером.
  
**Была ли эта уязвимость опубликована до выпуска этого бюллетеня безопасности?**    
Нет. Корпорация Майкрософт получила сведения об этой уязвимости из надежных источников.
  
**Получала ли корпорация Майкрософт к моменту выпуска этого бюллетеня по безопасности какие-либо сведения о том, что уязвимость была использована злоумышленниками?**    
Нет. Корпорация Майкрософт не получала никакой информации, указывающей на то, что до выпуска этого бюллетеня по безопасности данная уязвимость была открыто использована для организации атак на пользовательские компьютеры.

**Проверка успешного применения обновления**

-   **Анализатор безопасности Microsoft Baseline Security Analyzer**

    Для проверки успешного применения обновления безопасности можно воспользоваться средством Microsoft Baseline Security Analyzer (MBSA). Дополнительные сведения см. выше в разделе **"Руководство и средства по диагностике и развертыванию"** данного бюллетеня.

-   **Проверка версий файлов**

    Эти действия зависят от установленного выпуска Microsoft Windows. В случае затруднений для выполнения процедуры обратитесь к документации по установленной версии продукта.

    1.  Нажмите кнопку **Пуск** и выберите пункт **Найти**.
    2.  На панели **Результаты поиска** в области **Помощник по поиску** выберите пункт **Файлы и папки**.
    3.  В поле **Часть имени или имя файла целиком** введите имя файла из соответствующей справочной таблицы и нажмите **Найти**.
    4.  В списке файлов щелкните правой кнопкой мыши имя файла из соответствующей таблицы сведений о файлах и выберите **Свойства**.
        **Примечание**. В зависимости от выпуска операционной системы или имеющихся на компьютере программ некоторые файлы из таблицы сведений о файлах могут быть не установлены.
    5.  На вкладке **Версия** определите версию файла, установленного на компьютере, сравнив ее с версией, приведенной в таблице сведений о файлах.
        **Примечание**. Все атрибуты, кроме версии файла, в ходе установки могут изменяться. Проверка того, что обновление было применено, путем сравнения других атрибутов файла с данными из таблицы сведений о файлах не поддерживается. Кроме того, при определенных условиях файлы в ходе установки могут быть переименованы. Если сведения о файле или версии отсутствуют, обратитесь к другим доступным методам проверки установки обновления.

-   **Проверка параметров системного реестра**

    Файлы, установленные этим обновлением для системы безопасности, также можно проверить в разделах реестра, перечисленных в **справочной таблице** данного раздела.

    Эти разделы реестра могут не содержать полного списка установленных файлов. Кроме того, эти разделы реестра могут быть созданы неправильно, если администратор или сборщик систем интегрируют или добавляют данное обновление для системы безопасности в исходные установочные файлы Windows.

#### Сведения о развертывании
  
**Установка обновления**
  
При установке данного обновления для системы безопасности программа-установщик проверяет, не выполнялось ли ранее обновление одного или нескольких обновляемых файлов с помощью исправления Майкрософт.
  
Если один из файлов был обновлен ранее другим исправлением, установщик копирует на компьютер файлы RTMQFE, SP1QFE и SP2QFE. В противном случае будут скопированы файлы RTMGDR, SP1GDR или SP2GDR. Обновления для системы безопасности могут не содержать всех этих вариантов. Дополнительную информацию о поведении такого рода см. в [статье 824994 базы знаний Майкрософт](http://support.microsoft.com/kb/824994).
  
Дополнительные сведения об установщике см. на [веб-сайте Microsoft TechNet](http://go.microsoft.com/fwlink/?linkid=38951).
  
Подробнее о терминологии, встречающейся в этом бюллетене, например о том, что такое *исправление*, см. в [статье 824684 базы знаний Майкрософт](http://support.microsoft.com/kb/824684).

**Проверка успешного применения обновления**

-   **Программа Microsoft Baseline Security Analyzer (MBSA)**

    Для проверки успешного применения обновления безопасности можно воспользоваться средством Microsoft Baseline Security Analyzer (MBSA). Дополнительные сведения см. выше в разделе **"Руководство и средства по диагностике и развертыванию"** данного бюллетеня.

-   **Проверка версий файлов**

    Эти действия зависят от установленного выпуска Microsoft Windows. В случае затруднений для выполнения процедуры обратитесь к документации по установленной версии продукта.

    1.  Нажмите кнопку **Пуск** и выберите пункт **Найти**.
    2.  На панели **Результаты поиска** в области **Помощник по поиску** выберите пункт **Файлы и папки**.
    3.  В поле **Часть имени или имя файла целиком** введите имя файла из соответствующей справочной таблицы и нажмите **Найти**.
    4.  В списке файлов щелкните правой кнопкой мыши имя файла из соответствующей таблицы сведений о файлах и выберите **Свойства**.
        **Примечание**. В зависимости от выпуска операционной системы или имеющихся на компьютере программ некоторые файлы из таблицы сведений о файлах могут быть не установлены.
    5.  На вкладке **Версия** определите версию файла, установленного на компьютере, сравнив ее с версией, приведенной в таблице сведений о файлах.
        **Примечание**. Все атрибуты, кроме версии файла, в ходе установки могут изменяться. Проверка того, что обновление было применено, путем сравнения других атрибутов файла с данными из таблицы сведений о файлах не поддерживается. Кроме того, при определенных условиях файлы в ходе установки могут быть переименованы. Если сведения о файле или версии отсутствуют, обратитесь к другим доступным методам проверки установки обновления.

-   **Проверка параметров системного реестра**

    Файлы, установленные этим обновлением для системы безопасности, также можно проверить в разделах реестра, перечисленных в **справочной таблице** данного раздела.

    Эти разделы реестра могут не содержать полного списка установленных файлов. Кроме того, эти разделы реестра могут быть созданы неправильно, если администратор или сборщик систем интегрируют или добавляют данное обновление для системы безопасности в исходные установочные файлы Windows.
