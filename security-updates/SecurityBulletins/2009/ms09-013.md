Security Bulletin

Бюллетень по безопасности (Майкрософт) MS09-013 - Критическое
=============================================================

Уязвимости в службах Windows HTTP делают возможным удаленное выполнение кода (960803)
-------------------------------------------------------------------------------------

Дата публикации: 14 апреля 2009 г. | Дата обновления: 29 апреля 2009 г.

**Версия:** 1.1

### Общие сведения

#### Аннотация

Это обновление для системы безопасности устраняет уязвимости служб Microsoft Windows HTTP (WinHTTP). Об одной из них сообщалось в открытых источниках, а о двух других сведения предоставлялись в частном порядке. Наиболее существенная из них делает возможным удаленное выполнение кода. Воспользовавшись этой уязвимостью, злоумышленник может захватить полный контроль над системой, что позволит ему устанавливать программы, просматривать, изменять и уничтожать данные или создавать новые учетные записи с неограниченными полномочиями. Риск для пользователей, учетные записи которых имеют ограниченные права, меньше, чем для пользователей, работающих с правами администратора.

Это обновление для системы безопасности является критическим для всех поддерживаемых выпусков Microsoft Windows 2000, Windows XP, Windows Server 2003, Windows Vista и Windows Server 2008. Дополнительные сведения см. в подразделе **Подвержены и не подвержены уязвимости** этого раздела.

Оно устраняет уязвимости, изменяя способ, которым службы Windows HTTP обрабатывают ошибки и проверяют сертификаты, и обеспечивая правильное использование этими службами механизма защиты отображения учетных данных NTLM. Дополнительные сведения об этих уязвимостях см. в подразделе "Вопросы и ответы" для каждой отдельной уязвимости в разделе **Сведения об уязвимостях**.

**Рекомендация.** У большинства клиентов включено автоматическое обновление, поэтому нет необходимости предпринимать какие-либо действия: данное обновление для системы безопасности загрузится и установится автоматически. Клиентам, у которых не включено автоматическое обновление, необходимо проверить наличие обновлений и установить это обновление вручную. Дополнительные сведения об особых параметрах конфигурации автоматического обновления см. в [статье 294871 базы знаний Майкрософт](http://support.microsoft.com/kb/294871).

При установке на предприятиях, а также администраторам или конечным пользователям, которые хотят применить данное обновление для системы безопасности вручную, корпорация Майкрософт рекомендует сделать это немедленно, используя программное обеспечение для управления обновлениями или проверив его наличие в [Центре обновления Майкрософт](http://go.microsoft.com/fwlink/?linkid=40747).

См. также раздел **Руководство и средства по диагностике и развертыванию** этого бюллетеня.

**Известные проблемы.** В [статье 960803 базы знаний Майкрософт](http://support.microsoft.com/kb/960803) описаны известные на данный момент проблемы, с которыми можно столкнуться при установке этого обновления для системы безопасности. В этой статье также приводятся решения, рекомендованные для устранения этих проблем.

Вопросы и ответы о данном обновлении безопасности
-------------------------------------------------

<span></span>
**Почему этот бюллетень был исправлен 29 апреля 2009 г.?**  
В бюллетене указывается, что в раздел **Известные проблемы, связанные с этим обновлением для системы безопасности**[статьи 960803 базы знаний Майкрософт](http://support.microsoft.com/kb/960803) добавлены сведения о проблеме совместимости приложений после установки обновления в системах с агентом входа Websense, настроенным на использование проверки подлинности Windows с запросом и подтверждением (NTLM).

**Где находятся дополнительные сведения о файлах?**  
Дополнительные сведения о файлах см. в [статье 960803 базы знаний Майкрософт](http://support.microsoft.com/kb/960803).

**Подвержены ли этой уязвимости бета-версии систем Windows Server 2008 с пакетом обновления 2 (SP2), Windows Vista с пакетом обновления 2 (SP2) и Windows 7?**
Да. Об этих уязвимостях было объявлено после выпуска бета-версий перечисленных систем. Их пользователям рекомендуется загрузить и установить это обновление.

Обновления для системы безопасности доступны в [Центре обновления Майкрософт](http://go.microsoft.com/fwlink/?linkid=40747) и [Центре обновления Windows](http://go.microsoft.com/fwlink/?linkid=21130).

**Какое отношение имеет данное обновление к обновлению MS09-014?**  
Уязвимость служб Windows HTTP, связанная с отображением учетных данных ([CVE-2009-0550](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2009-0550)), которая описана в этом бюллетене по безопасности, также затрагивает браузер Internet Explorer и программный интерфейс Windows Internet (WinInet). Корпорация Майкрософт рекомендует пользователям браузера Internet Explorer установить обновление [MS09-014](http://go.microsoft.com/fwlink/?linkid=146659), содержащееся в накопительном обновлении для системы безопасности Internet Explorer (963027).

**С какими известными проблемами могут столкнуться пользователи при установке этого обновления для системы безопасности?**  
В [статье 960803 базы знаний Майкрософт](http://support.microsoft.com/kb/960803) описаны известные на данный момент проблемы, с которыми можно столкнуться при установке этого обновления для системы безопасности. В этой статье также приводятся решения, рекомендованные для устранения этих проблем.

**Почему это обновление устраняет несколько уязвимостей системы безопасности, о которых уже сообщалось?**  
Это обновление разработано для нескольких уязвимостей, поскольку для устранения их причин требуется выполнить изменения в связанных файлах. Вместо установки нескольких почти одинаковых обновлений пользователям необходимо установить только данное обновление.

**Производит ли данное обновление какие-либо изменения в функциях системы, связанные с безопасностью?**  
Да. В целях решения проблемы, описанной в CVE-2009-0089, корпорация Майкрософт внедрила новый раздел реестра UseFQDNFallbackForCnCompare в раздел SOFTWARE\\Microsoft\\Windows\\CurrentVersion\\Internet Settings\\WinHttp, который контролирует поведение службы WinHTTP в ходе сверки полного доменного имени узла с записью различающегося имени сертификата.

По умолчанию этого раздела не существует, что обеспечивает полную проверку службой WinHTTP соответствия различающегося имени, которое содержится в сертификате, различающемуся имени узла, к которому выполняется подключение. Старый режим работы службы можно включить повторно, задав для этого раздела реестра значение True. В таком случае проверка будет менее строгой и будет допускать соответствие сертификата изменению имени путем пересылки на уровень DNS. Этот режим обеспечивает более низкий уровень безопасности, который является стандартным до установки данного обновления для системы безопасности.

**На моем компьютере установлена более ранняя версия программного обеспечения, описанного в этом бюллетене по безопасности. Что мне следует сделать?**  
Продукты, перечисленные в этом бюллетене, проверены на наличие уязвимости в тех или иных версиях. Жизненные циклы поддержки прочих версий программного обеспечения истекли. Сведения о жизненных циклах поддержки версий используемых программных продуктов можно найти на веб-узле [Microsoft Support Lifecycle](http://go.microsoft.com/fwlink/?linkid=21742).

Пользователям более ранних версий программного обеспечения следует срочно перейти на поддерживаемые версии, чтобы снизить вероятность наличия в системе уязвимостей. Дополнительные сведения о жизненном цикле продуктов Windows см. на веб-узле [Правила по срокам поддержки продуктов Microsoft](http://go.microsoft.com/fwlink/?linkid=21742). Дополнительные сведения о продленном периоде поддержки обновлений для системы безопасности данных версий или выпусков продуктов можно найти на веб-узле [службы технической поддержки Microsoft](http://go.microsoft.com/fwlink/?linkid=33328) (на английском языке).

Для получения сведений о возможных вариантах обслуживания более ранних версий программного обеспечения свяжитесь с представителем группы по работе с заказчиками корпорации Майкрософт, менеджером по технической поддержке или представителем соответствующей партнерской компании. Пользователи, у которых нет договора типа Alliance, Premier или Authorized, могут обратиться в местное представительство корпорации Майкрософт. Для получения контактной информации посетите веб-узел [Microsoft Worldwide Information](http://go.microsoft.com/fwlink/?linkid=33329) (на английском языке) выберите страну и нажмите кнопку **Go** (Перейти), чтобы вывести список телефонных номеров. Дозвонившись, попросите связать вас с менеджером по продажам службы поддержки Premier. Дополнительную информацию можно получить в разделе[Windows Operating System Product Support Lifecycle FAQ](http://go.microsoft.com/fwlink/?linkid=33330).

Уязвимость служб Windows HTTP, связанная с потерей точности целых чисел, — CVE-2009-0086  
----------------------------------------------------------------------------------------
  
<span></span>
В способе, которым службы Windows HTTP обрабатывают определенные значения, возвращаемые удаленным веб-сервером, существует уязвимость, делающая возможным удаленное выполнение кода. Воспользовавшись этой уязвимостью, злоумышленник может захватить полный контроль над системой, Он сможет устанавливать программы, просматривать, изменять и удалять данные или создавать учетные записи с такими же правами, как и у службы или приложения, которое вызывает программный интерфейс WinHTTP, с целью подключения к веб-серверу злоумышленника.
  
В перечне Common Vulnerabilities and Exposures этой уязвимости присвоен номер [CVE-2009-0086](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2009-0086).
  
#### Факторы, снижающие опасность уязвимости служб Windows HTTP, связанной с потерей точности целых чисел, — CVE-2009-0086
  
К факторам, снижающим опасность уязвимости, относятся параметр, стандартная конфигурация или общие рекомендации, которые при использовании по умолчанию могут снизить опасность использования уязвимости. Следующие факторы могут снизить опасность использования уязвимости.
  
-   Все поддерживаемые выпуски Internet Explorer не подвержены этой уязвимости, так как службы Windows HTTP не используются данным программным обеспечением.
  
#### Методы обхода уязвимости служб Windows HTTP, связанной с потерей точности целых чисел, — CVE-2009-0086
  
Корпорацией Майкрософт не найдены методы обхода этой уязвимости.
  
#### Вопросы и ответы об уязвимости служб Windows HTTP, связанной с потерей точности целых чисел, — CVE-2009-0086
  
**Какова область воздействия данной уязвимости?**    
Эта уязвимость связана с запуском программного кода в удаленном режиме. Воспользовавшись этой уязвимостью, злоумышленник может захватить полный контроль над системой, Он сможет устанавливать программы, просматривать, изменять и удалять данные или создавать учетные записи с такими же правами, как и у службы или приложения, которое вызывает программный интерфейс WinHTTP, с целью подключения к веб-серверу злоумышленника.
  
Данной уязвимости подвержены компоненты Windows и приложения сторонних производителей, которые используют службы Windows HTTP и поддерживают подключение к удаленным ненадежным веб-серверам.
  
**В чем причина уязвимости?** Уязвимость вызвана недостаточным уровнем проверки параметра, возвращаемого удаленным веб-сервером, что может приводить к выполнению произвольного кода.
  
**Что представляют собой службы Windows HTTP?**    
Службы Windows HTTP (WinHTTP) обеспечивают прикладной программный интерфейс (API) клиента HTTP для отправки запросов на веб-серверы по протоколу HTTP. Службы WinHTTP могут использоваться как компонентами Microsoft Windows, так и программным обеспечением сторонних производителей. Наряду с другими они также используются службой Universal Plug and Play (UPNP).
  
**Что такое Universal Plug and Play?**    
Функция Universal Plug and Play (UPnP) позволяет устройствам в сети обнаруживать другие устройства и определять способ взаимодействия с ними. Термин UPnP проще понять, проведя аналогию с функцией самонастраивающихся устройств (plug-and-play, PnP), которая хорошо знакома большинству пользователей Windows. Она позволяет операционной системе обнаруживать новое подключаемое оборудование. Например, при подключении новой мыши функция PnP позволяет системе Windows определить новое устройство, загрузить необходимые драйверы и начать его использование. Функция UPnP расширяет применение этого принципа до устройств в сети, в отличие от локальной системы. Она позволяет компьютерам обнаруживать другие устройства в сети и определять способ взаимодействия с ними. Например, компьютер может использовать функцию UPnP для определения наличия сетевых принтеров, которые он может использовать, и получить данные о том, как с ними взаимодействовать.
  
**Что представляет собой протокол SSDP (Simple Service Discovery Protocol)?**    
Обнаружение устройств — это процесс, с помощью которого компьютеры с поддержкой UPnP получают сведения о доступности устройств, которые они могут использовать, а также о способе запроса обслуживания у этих устройств. При загрузке компьютера с поддержкой функции UPnP в сети уже могут присутствовать устройства, которые он может использовать. Чтобы выяснить это, компьютер отправляет широковещательный запрос (называемый директивой M-SEARCH). У устройств с поддержкой UPnP в подсети запрашивается непосредственный ответ на запрос, а также сведения о способе доступа к ним.
  
Аналогично, когда загружается устройство с поддержкой UPnP (например, UPnP-принтер), в сети уже могут присутствовать компьютеры, поддерживающие UPnP, которые могли бы использовать это устройство. Устройство отправляет сообщение (называемое директивой NOTIFY) всем компьютерам в подсети, сообщая о своем присутствии и готовности обслуживать их.
  
**Что может сделать злоумышленник, который воспользуется этой уязвимостью?**    
Воспользовавшись этой уязвимостью, злоумышленник может получить те же права пользователя, что и служба или приложение, вызывающее API WinHTTP, с целью подключения к своему вредоносному веб-северу. Приложения, выполняемые в учетных записях с ограниченными правами пользователя, могут быть подвержены уязвимости меньше, чем приложения, выполняемые в учетных записях с правами администратора.
  
**Каким образом злоумышленник может воспользоваться этой уязвимостью?**    
Чтобы воспользоваться этой уязвимостью, злоумышленник должен сделать так, чтобы программа, которая использует службы Windows HTTP, подключилась к вредоносному веб-серверу. Следует отметить, что служба UPnP использует библиотеки WinHTTP. Если она включена, злоумышленники в локальной подсети могут отвечать на запросы SSDP и вынуждать службу UPnP выполнять подключение с помощью WinHTTP к вредоносному узлу, который затем может использовать эту уязвимость. Служба UPNP выполняется в учетной записи LocalService, которая имеет ограниченные права в системе.
  
**Какие системы в первую очередь подвергаются риску?**    
В первую очередь риску воздействия подвергаются рабочие станции и серверы терминалов. Серверы могут быть подвержены большему риску, если администраторы позволяют пользователям входить в систему на сервере и запускать программы. Однако общепринятой практикой настоятельно рекомендуется не предоставлять пользователям таких полномочий.
  
**Как действует обновление?**    
Обновление устраняет эту уязвимость, изменяя способ, которым службы Windows HTTP обрабатывают позволяющую использовать ее ошибку.
  
**Было ли объявлено о данной уязвимости до выпуска этого бюллетеня по безопасности?**    
Нет. Корпорация Майкрософт получила сведения об этой уязвимости из достоверных источников.
  
**Получала ли корпорация Майкрософт к моменту выпуска этого бюллетеня по безопасности какие-либо сведения о том, что уязвимость была использована злоумышленниками?**    
Нет. Корпорация Майкрософт не получала никакой информации, указывающей на то, что эта уязвимость была открыто использована для атак на пользователей, и ей не были представлены никакие доказательства публикации концептуального кода до первоначального выпуска этого бюллетеня безопасности.
  
Уязвимость служб Windows HTTP, связанная с несоответствием имени сертификата, — CVE-2009-0089

<span></span>
В службах Windows HTTP существует уязвимость, которая делает возможной спуфинг. Она связана с неполной проверкой различающегося имени в цифровом сертификате. В сочетании с определенными атаками, например со спуфингом DNS, эта уязвимость может допускать подделку цифрового сертификата веб-сайта для любого приложения, использующего службы Windows HTTP.
  
В перечне Common Vulnerabilities and Exposures этой уязвимости присвоен номер [CVE-2009-0089](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2009-0089).
  
#### Факторы, снижающие опасность уязвимости служб Windows HTTP, связанной с несоответствием имени сертификата, — CVE-2009-0089
  
К факторам, снижающим опасность уязвимости, относятся параметр, стандартная конфигурация или общие рекомендации, которые при использовании по умолчанию могут снизить опасность использования уязвимости. Следующие факторы могут снизить опасность использования уязвимости.
  
-   Все поддерживаемые выпуски Internet Explorer не подвержены этой уязвимости, так как службы Windows HTTP не используются данным программным обеспечением.
  
#### Методы обхода уязвимости служб Windows HTTP, связанной с несоответствием имени сертификата, — CVE-2009-0089
  
Корпорацией Майкрософт не найдены методы обхода этой уязвимости.
  
#### Вопросы и ответы об уязвимости служб Windows HTTP, связанной с несоответствием имени сертификата, — CVE-2009-0089
  
**Какова область воздействия данной уязвимости?**    
Это уязвимость спуфинга, которая существует в службах Windows HTTP. В сочетании с определенными атаками, например со спуфингом DNS, эта уязвимость может допускать подделку цифрового сертификата веб-сайта для любого приложения, использующего службы Windows HTTP.
  
**В чем причина уязвимости?** Когда приложение вызывает службы Windows HTTP для установки подключения к удаленному веб-серверу, службы сверяют с сертификатом исходного веб-сайта, к которому пытается подключиться узел, только полное доменное имя URL-адреса. Определенная комбинация спуфинга DNS может перенаправить подключение на другой веб-сервер, имеющий действительный сертификат для данного узла, но не для веб-страницы, подключение к которой было инициировано службами WinHTTP. Сочетание спуфинга DNS и этой уязвимости может являться причиной ошибочного определения службами WinHTTP сертификата удаленного веб-сервера в качестве приемлемого. Поскольку службы WinHTTP взаимодействуют с пользователем посредством приложения, он может не получать от приложения предупреждение о таком расхождении.
  
**Что представляют собой службы Windows HTTP?**    
Службы Windows HTTP (WinHTTP) обеспечивают прикладной программный интерфейс (API) клиента HTTP для отправки запросов на HTTP-серверы по протоколу HTTP.
  
**Что такое цифровой сертификат?**    
Цифровой сертификат — это типичный вид учетных данных, который служит для проверки подлинности удостоверений. Надежные организации предоставляют сертификаты отдельным пользователям или компаниям, называемым субъектами сертификата. Надежные организации, выпускающие сертификаты, называются центрами сертификации (ЦС) или издателями сертификата. Надежный центр сертификации выпускает сертификат только после проверки удостоверения субъекта сертификата.
  
**Что такое спуфинг DNS?**    
Служба доменных имен (DNS) является одним из наборов протоколов, признанных отраслевым стандартом, в который входят протоколы TCP/IP. Служба DNS реализована с использованием двух программных компонентов: DNS-сервера и DNS-клиента (или распознавателя). Оба компонента работают в качестве фоновых приложений служб. Сетевые ресурсы обозначаются числовыми IP-адресами, но такие адреса сложны для запоминания пользователями сети. В базе данных службы DNS содержатся записи, которые сопоставляют понятные пользователю буквенно-цифровые имена сетевых ресурсов с IP-адресами, используемыми этими ресурсами для связи друг с другом. Таким образом, служба DNS выполняет роль символического устройства, позволяющего пользователям легко запоминать имена сетевых ресурсов. Дополнительные сведения и логические схемы взаимодействия службы DNS с другими технологиями Windows см. в статье [Что такое DNS?](http://technet.microsoft.com/en-us/library/cc787921.aspx) (на английском языке) на веб-сайте TechNet.
  
Спуфингом DNS называют широкий спектр атак, в которых в одной точке системы DNS неправильные ответы DNS вводятся третьей стороной, обычно не ответственной за них. Атака такого типа может приводить к разрешению имени домена или узла в IP-адрес, не предназначенный для разрешения владельцами этого имени. Таким образом узел можно перенаправить на сервер третьей стороны, который может оказаться вредоносным.
  
**Что может сделать злоумышленник, который воспользуется этой уязвимостью?**    
Злоумышленник может олицетворить надежный веб-сайт и предоставить приложению вредоносное содержимое с помощью служб HTTP Windows, которые будут определять содержимое так, как если бы оно поступило с надежного веб-сайта.
  
**Каким образом злоумышленник может воспользоваться этой уязвимостью?**    
Чтобы направить исходящее подключение на контролируемый им сервер, злоумышленнику требуется использовать эту уязвимость в сочетании со спуфингом DNS или аналогичной атакой. Затем он сможет предоставить сертификат, действительный для его веб-сайта, подключение к которому будет неправильно определено службами Windows HTTP как действительное для узла назначения веб-сайта, к которому изначально пытался подключиться пользователь.
  
**Какие системы в первую очередь подвергаются риску?**    
В первую очередь риску воздействия подвергаются рабочие станции и серверы терминалов. Серверы могут быть подвержены большему риску, если администраторы позволяют пользователям входить в систему на сервере и запускать программы. Однако общепринятой практикой настоятельно рекомендуется не предоставлять пользователям таких полномочий.
  
**Как действует обновление?**    
Оно устраняет уязвимость, изменяя способ, которым службы Windows HTTP проверяют сертификаты, принадлежащие надежным веб-сайтам.
  
**Было ли объявлено о данной уязвимости до выпуска этого бюллетеня по безопасности?**    
Нет. Корпорация Майкрософт получила сведения об этой уязвимости из достоверных источников.
  
**Получала ли корпорация Майкрософт к моменту выпуска этого бюллетеня по безопасности какие-либо сведения о том, что уязвимость была использована злоумышленниками?**    
Нет. Корпорация Майкрософт не получала никакой информации, указывающей на то, что эта уязвимость была открыто использована для атак на пользователей, и ей не были представлены никакие доказательства публикации концептуального кода до первоначального выпуска этого бюллетеня безопасности.

---------------------------------------------------------------------------------------
  
<span></span>
В способе, которым службы Windows HTTP обрабатывают учетные данные NTLM при подключении пользователя к веб-серверу злоумышленника, существует уязвимость, делающая возможным удаленное выполнение кода. Она позволяет злоумышленнику получить учетные данные пользователя и выполнить код в контексте пользователя, вошедшего в систему. Если пользователь вошел в систему с правами администратора, то злоумышленник, которому удалось воспользоваться уязвимостью, может установить полный контроль над системой. что позволит ему устанавливать программы, просматривать, изменять и уничтожать данные или создавать новые учетные записи с неограниченными полномочиями. Риск для пользователей, учетные записи которых имеют ограниченные права, меньше, чем для пользователей, работающих с правами администратора.
  
В перечне Common Vulnerabilities and Exposures этой уязвимости присвоен номер [CVE-2008-2550](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2009-0550).
  
#### Факторы, снижающие опасность уязвимости служб Windows HTTP, связанной с отображением учетных данных, — CVE-2009-0550
  
К факторам, снижающим опасность уязвимости, относятся параметр, стандартная конфигурация или общие рекомендации, которые при использовании по умолчанию могут снизить опасность использования уязвимости. Следующие факторы могут снизить опасность использования уязвимости.
  
-   В случае атаки через Интернет злоумышленник должен владеть веб-узлом, который содержит веб-страницу, предназначенную для использования данной уязвимости. Кроме того, веб-узлы, которые подверглись атаке злоумышленника и веб-узлы, которые принимают или размещают сведения или рекламу, могут иметь специальное содержимое, использующее данную уязвимость. В любом случае злоумышленник не может заставить пользователей посетить эти веб-узлы. Он старается склонить их к этому, убеждая, как правило, щелкнуть ссылку, ведущую на веб-узел, в сообщении электронной почты или программы обмена мгновенными сообщениями.  
-   Злоумышленник, успешно воспользовавшийся данной уязвимостью, может получить те же самые права, которые имеет локальный пользователь. Риск для пользователей, учетные записи которых имеют ограниченные права, меньше, чем для пользователей, работающих с правами администратора.  
-   Во всех поддерживаемых выпусках Windows Vista и Windows Server 2008 параметр для профиля брандмауэра Windows по умолчанию имеет значение "Общий". Это позволяет снизить подверженность этих платформ данной уязвимости, поскольку такая настройка профиля подразумевает блокирование брандмауэром трафика SMB. Поскольку для проведения атаки и отображения учетных данных необходимо, чтобы на клиенте работала служба, поддерживающая проверку подлинности NTLM, наиболее вероятным направлением атаки будет являться SMB.

Корпорацией Майкрософт не найдены методы обхода этой уязвимости.
  
#### Вопросы и ответы об уязвимости служб Windows HTTP, связанной с отображением учетных данных, — CVE-2009-0550
  
**Какова область воздействия данной уязвимости?**    
Эта уязвимость связана с запуском программного кода в удаленном режиме. Воспользовавшись этой уязвимостью, злоумышленник может захватить полный контроль над системой, что позволит ему устанавливать программы, просматривать, изменять и уничтожать данные или создавать новые учетные записи с неограниченными полномочиями.
  
Кроме того, эта проблема затрагивает браузер Internet Explorer и API WinInet. Корпорация Майкрософт рекомендует пользователям браузера Internet Explorer установить обновление [MS09-014](http://go.microsoft.com/fwlink/?linkid=146659), содержащееся в накопительном обновлении для системы безопасности Internet Explorer (963027).
  
**В чем причина уязвимости?** Службы Windows HTTP неправильно выбирают средства защиты отображения учетных данных NTLM для обеспечения того, что учетные данные не будут получены и использованы в ущерб пользователю.
  
**Что представляют собой службы Windows HTTP?**    
Службы Windows HTTP (WinHTTP) обеспечивают прикладной программный интерфейс (API) клиента HTTP для отправки запросов на HTTP-серверы по протоколу HTTP.
  
**Что может сделать злоумышленник, который воспользуется этой уязвимостью?**    
Злоумышленник может получить права пользователя, вошедшего в систему, и выполнять любые действия, которые позволяют полномочия пользователя, вошедшего в систему. Если пользователь вошел в систему с правами администратора, то злоумышленник, которому удалось воспользоваться уязвимостью, может установить полный контроль над системой. что позволит ему устанавливать программы, просматривать, изменять и уничтожать данные или создавать новые учетные записи с неограниченными полномочиями. Риск для пользователей, учетные записи которых имеют ограниченные права, меньше, чем для пользователей, работающих с правами администратора.
  
**Каким образом злоумышленник может воспользоваться этой уязвимостью?**    
Для использования уязвимости необходимо, чтобы пользователь получил доступ к вредоносному серверу посредством уязвимой версии служб Windows HTTP. Злоумышленнику требуется разместить специальный веб-сервер, который будет перехватывать пользовательские учетные данные NTLM для входа в систему. Затем он может воспроизвести эти учетные данные для службы в клиентской системе пользователя, которая поддерживает проверку подлинности NTLM. У злоумышленника нет возможности заставить пользователя посетить специальный сервер. Однако он может склонить их к этому, убедив, как правило, щелкнуть ссылку в сообщении электронной почты или в запросе программы обмена мгновенными сообщениями.
  
**Какие системы в первую очередь подвергаются риску?**    
В первую очередь риску воздействия подвергаются рабочие станции и серверы терминалов. Серверы могут быть подвержены большему риску, если администраторы позволяют пользователям входить в систему на сервере и запускать программы. Однако общепринятой практикой настоятельно рекомендуется не предоставлять пользователям таких полномочий.
  
**Как действует обновление?**    
Обновление устраняет уязвимость, обеспечивая правильный выбор службами Windows HTTP подходящего механизма защиты учетных данных Windows.
  
**Почему этой уязвимости и уязвимости WinINet, связанной с отображением учетных данных, присвоены одинаковые номера CVE (CVE-2009-0550) в бюллетене MS09-014, "Накопительное обновление для системы безопасности браузера (963027)"?**    
Уязвимость CVE-2009-0550 воздействует на WinINet, как используемый браузером Internet Explorer, так и службами Windows HTTP, которые являются компонентом системы Windows. Корпорация Майкрософт рекомендует пользователям браузера Internet Explorer установить обновление [MS09-014](http://go.microsoft.com/fwlink/?linkid=146659), содержащееся в накопительном обновлении для системы безопасности Internet Explorer (963027).
  
**Было ли объявлено о данной уязвимости до выпуска этого бюллетеня по безопасности?**    
Да. О данной уязвимости сообщалось в открытых источниках. В перечне Common Vulnerabilities and Exposures данной уязвимости присвоен номер [CVE-2009-0550](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2009-0550).
  
**Получала ли корпорация Майкрософт к моменту выпуска этого бюллетеня по безопасности какие-либо сведения о том, что уязвимость была использована злоумышленниками?**    
Нет. Корпорация Майкрософт не получала никакой информации, указывающей на то, что эта уязвимость была открыто использована для атак на пользователей, и ей не были представлены никакие доказательства публикации концептуального кода до первоначального выпуска этого бюллетеня безопасности.

#### Сведения о развертывании

**Установка обновления**

При установке этого обновления для системы безопасности проверяется, не выполнялось ли обновление этого файла (файлов) ранее с помощью исправления Microsoft.

Если один из файлов был обновлен ранее другим исправлением, установщик копирует на компьютер файлы RTMQFE, SP1QFE и SP2QFE. В противном случае будут скопированы файлы RTMGDR, SP1GDR или SP2GDR. Обновления для системы безопасности могут не содержать всех этих вариантов. Дополнительную информацию о поведении такого рода см. в [статье 824994 базы знаний Майкрософт](http://support.microsoft.com/kb/824994).

Дополнительные сведения об установщике см. на [веб-узле Microsoft TechNet](http://go.microsoft.com/fwlink/?linkid=38951).

Для получения подробных сведений о терминологии, встречающейся в этом бюллетене, например, *исправление*, обратитесь к [статье 824684 базы знаний Майкрософт](http://support.microsoft.com/kb/824684).

**Проверка успешного применения обновления**

-   **Программа Microsoft Baseline Security Analyzer**

    Для проверки успешного применения обновления безопасности можно воспользоваться средством Microsoft Baseline Security Analyzer (MBSA). Дополнительные сведения см. выше в разделе **"Руководство и средства по диагностике и развертыванию"** данного бюллетеня.

-   **Проверка версий файлов**

    Эти действия зависят от установленного выпуска Microsoft Windows. В случае затруднений для выполнения процедуры обратитесь к документации по установленной версии продукта.

    1.  Нажмите кнопку **Пуск** и выберите пункт **Найти**.
    2.  На панели **Результаты поиска** в области **Помощник по поиску** выберите пункт **Файлы и папки**.
    3.  В области **Часть имени или имя файла целиком** введите имя файла из соответствующей справочной таблицы по файлам и нажмите кнопку **Найти**.
    4.  В списке файлов щелкните правой кнопкой мыши имя файла из соответствующей таблицы сведений о файлах и выберите **Свойства**.
        **Примечание**. В зависимости от выпуска операционной системы или имеющихся на компьютере программ некоторые файлы из таблицы сведений о файлах могут быть не установлены.
    5.  На вкладке **Версия** определите версию файла, установленного на компьютере, сравнив ее с версией, приведенной в таблице сведений о файлах.
        **Примечание**. Все атрибуты, кроме версии файла, в ходе установки могут изменяться. Проверка того, что обновление было применено, путем сравнения других атрибутов файла с данными из таблицы сведений о файлах не поддерживается. Кроме того, при определенных условиях файлы в ходе установки могут быть переименованы. Если сведения о файле или версии отсутствуют, обратитесь к другим доступным методам проверки установки обновления.

-   **Проверка параметров системного реестра**

    Файлы, установленные этим обновлением для системы безопасности, также можно проверить в разделах реестра, перечисленных в **справочной таблице** данного раздела.

    Эти разделы реестра могут не содержать полного списка установленных файлов. Кроме того, эти разделы реестра могут быть созданы неправильно, если администратор или сборщик систем интегрируют или добавляют данное обновление для системы безопасности в исходные установочные файлы Windows.

**Проверка успешного применения обновления**

-   **Программа Microsoft Baseline Security Analyzer**

    Для проверки успешного применения обновления безопасности можно воспользоваться средством Microsoft Baseline Security Analyzer (MBSA). Дополнительные сведения см. выше в разделе **"Руководство и средства по диагностике и развертыванию"** данного бюллетеня.

-   **Проверка версий файлов**

    Эти действия зависят от установленного выпуска Microsoft Windows. В случае затруднений для выполнения процедуры обратитесь к документации по установленной версии продукта.

    1.  Нажмите кнопку **Пуск** и выберите пункт **Найти**.
    2.  На панели **Результаты поиска** в области **Помощник по поиску** выберите пункт **Файлы и папки**.
    3.  В области **Часть имени или имя файла целиком** введите имя файла из соответствующей справочной таблицы по файлам и нажмите кнопку **Найти**.
    4.  В списке файлов щелкните правой кнопкой мыши имя файла из соответствующей таблицы сведений о файлах и выберите **Свойства**.
        **Примечание**. В зависимости от выпуска операционной системы или имеющихся на компьютере программ некоторые файлы из таблицы сведений о файлах могут быть не установлены.
    5.  На вкладке **Версия** определите версию файла, установленного на компьютере, сравнив ее с версией, приведенной в таблице сведений о файлах.
        **Примечание**. Все атрибуты, кроме версии файла, в ходе установки могут изменяться. Проверка того, что обновление было применено, путем сравнения других атрибутов файла с данными из таблицы сведений о файлах не поддерживается. Кроме того, при определенных условиях файлы в ходе установки могут быть переименованы. Если сведения о файле или версии отсутствуют, обратитесь к другим доступным методам проверки установки обновления.

-   **Проверка параметров системного реестра**

    Файлы, установленные этим обновлением для системы безопасности, также можно проверить в разделах реестра, перечисленных в **справочной таблице** данного раздела.

**Проверка успешного применения обновления**

-   **Программа Microsoft Baseline Security Analyzer**

    Для проверки успешного применения обновления безопасности можно воспользоваться средством Microsoft Baseline Security Analyzer (MBSA). Дополнительные сведения см. выше в разделе "Руководство и средства по диагностике и развертыванию" данного бюллетеня.

-   **Проверка версий файлов**

    Эти действия зависят от установленного выпуска Microsoft Windows. В случае затруднений для выполнения процедуры обратитесь к документации по установленной версии продукта.

    1.  Нажмите кнопку **Пуск** и выберите пункт **Найти**.
    2.  На панели **Результаты поиска** в области **Помощник по поиску** выберите пункт **Файлы и папки**.
    3.  В области **Часть имени или имя файла целиком** введите имя файла из соответствующей справочной таблицы по файлам и нажмите кнопку **Найти**.
    4.  В списке файлов щелкните правой кнопкой мыши имя файла из соответствующей таблицы сведений о файлах и выберите **Свойства**.
        **Примечание**. В зависимости от выпуска операционной системы или имеющихся на компьютере программ некоторые файлы из таблицы сведений о файлах могут быть не установлены.
    5.  На вкладке **Версия** определите версию файла, установленного на компьютере, сравнив ее с версией, приведенной в таблице сведений о файлах.
        **Примечание**. Все атрибуты, кроме версии файла, в ходе установки могут изменяться. Проверка того, что обновление было применено, путем сравнения других атрибутов файла с данными из таблицы сведений о файлах не поддерживается. Кроме того, при определенных условиях файлы в ходе установки могут быть переименованы. Если сведения о файле или версии отсутствуют, обратитесь к другим доступным методам проверки установки обновления.

-   **Проверка параметров системного реестра**

    Файлы, установленные этим обновлением для системы безопасности, также можно проверить в разделах реестра, перечисленных в **справочной таблице** данного раздела.

    Эти разделы реестра могут не содержать полного списка установленных файлов. Кроме того, эти разделы реестра могут быть созданы неправильно, если администратор или сборщик систем интегрируют или добавляют данное обновление для системы безопасности в исходные установочные файлы Windows.
