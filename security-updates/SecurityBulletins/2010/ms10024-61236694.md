Security Bulletin

Бюллетень по безопасности (Майкрософт) MS10-024 - Важное
========================================================

Уязвимости в Microsoft Exchange и службе SMTP Windows делают возможной атаку типа "отказ в обслуживании" (981832)
-----------------------------------------------------------------------------------------------------------------

Дата публикации: 13 апреля 2010 г. | Дата обновления: 13 июля 2010 г.

**Версия:** 2.0

### Общие сведения

#### Аннотация

Это обновление для системы безопасности устраняет одну опубликованную и одну обнаруженную пользователями уязвимость в Microsoft Exchange и службе SMTP Windows. Наиболее опасная из этих уязвимостей делает возможной атаку типа "отказ в обслуживании", если злоумышленник отправил специально созданный ответ DNS на компьютер со службой SMTP. По умолчанию компонент SMTP в системах Windows Server 2003, Windows Server 2003 x64 Edition и Windows XP Professional x64 Edition не устанавливается.

Это обновление для системы безопасности имеет уровень серьезности "существенный" для всех поддерживаемых выпусков Microsoft Windows 2000, Windows XP и Windows Server 2003, 32-разрядных и 64-разрядных (x64) выпусков Windows Server 2008, Windows Server 2008 R2 для компьютеров на базе x64-процессоров и Microsoft Exchange Server 2003. Это обновление имеет уровень серьезности "средний" для Microsoft Exchange Server 2000. Дополнительные сведения см. в подразделе **Подвержены и не подвержены уязвимости** далее в этом разделе.

Это обновление для системы безопасности устраняет уязвимость, исправляя способ, используемый компонентом SMTP для анализа записей MX и выделения памяти для интерпретации ответов на команды SMTP. Дополнительные сведения об этих уязвимостях см. в подразделе ответов на часто задаваемые вопросы для каждой уязвимости, упомянутой в разделе **Сведения об уязвимостях**.

**Рекомендация.** У большинства клиентов включено автоматическое обновление, поэтому нет необходимости предпринимать какие-либо действия: данное обновление для системы безопасности загрузится и установится автоматически. Клиентам, у которых не включено автоматическое обновление, необходимо проверить наличие обновлений и установить это обновление вручную. Дополнительные сведения об особых параметрах конфигурации автоматического обновления см. в [статье 294871 базы знаний Майкрософт](http://support.microsoft.com/kb/294871).

При установке на предприятиях, а также администраторам или конечным пользователям, которым необходимо установить данное обновление для системы безопасности вручную, корпорация Майкрософт рекомендует сделать это немедленно, используя программное обеспечение для управления обновлениями или проверив наличие обновлений через [службу обновлений Майкрософт](http://go.microsoft.com/fwlink/?linkid=40747).

См. также раздел **Руководство и средства по диагностике и развертыванию** этого бюллетеня.

**Известные проблемы.** В [статье 981832 базы знаний Майкрософт](http://support.microsoft.com/kb/981832) описаны известные на данный момент проблемы, с которыми пользователи могут столкнуться при установке этого обновления для системы безопасности. В этой статье также приводятся решения, рекомендованные для устранения этих проблем.

#### Подвержены и не подвержены уязвимости

Следующие продукты были проверены на наличие уязвимости в тех или иных версиях и выпусках. Прочие версии или выпуски не подвержены уязвимости, либо жизненные циклы их поддержки истекли. Сведения о жизненных циклах поддержки версий или выпусков используемых программных продуктов см. на веб-странице [сроков поддержки продуктов Майкрософт](http://go.microsoft.com/fwlink/?linkid=21742).

Часто задаваемые вопросы о данном обновлении безопасности
---------------------------------------------------------

<span></span>
**Почему этот бюллетень повторно опубликован 13 июля 2010 г.?**  
Корпорация Майкрософт повторно выпустила этот бюллетень, чтобы повторно предложить обновление для Windows Server 2008 и Windows Server 2008 R2 для устранения известной проблемы, описанной в KB976323; при установке этого обновления безопасности на компьютере под управлением Windows Server 2008, на котором установлен службы IIS (Internet Information Services), параметры конфигурации SMTP сбрасываются. При этом все параметры конфигурации SMTP, настроенные до установки этого обновления, будут потеряны. Вне необходимые параметры конфигурации SMTP следует снова установить вручную после установки этого обновления безопасности.

Повторный выпуск относится только к пакетам, связанным с KB976323. Пользователям, уже обновившим свои системы, повторно устанавливать это обновление не требуется.

**Почему выпущена новая редакция этого бюллетеня по безопасности от 15 апреля 2010 года?**  
Выпуск новой редакции этого бюллетеня по безопасности вызван тем, что в него добавлена информация об изменении обнаружения для Microsoft Exchange Server 2000 с пакетом обновления 3 (SP3). Это изменение затрагивает только способ обнаружения. Исправленный механизм обнаружения предлагает данное обновление для системы безопасности тем уязвимым системам, которым ранее оно не предлагалось. Этот бюллетень не вносит изменения в файлы обновления для системы безопасности. Если обновление KB976703 уже успешно установлено в этих системах, необходимости в его переустановке нет.

**Где находятся дополнительные сведения о файлах?**  
Местонахождение дополнительных сведений о файлах см. в справочных таблицах раздела **Развертывание обновления для системы безопасности**.

**Чем обновления для Microsoft Exchange Server 2007 и Microsoft Exchange Server 2010 отличаются от обновлений для Microsoft Exchange Server 2000 и Microsoft Exchange Server 2003?**  
С выпуском Microsoft Exchange 2007 обновления будут предоставляться по-новому. Сведения о схеме предоставления обновлений для Microsoft Exchange см. в статье TechNet [Предоставление обновлений для Exchange 2007](http://go.microsoft.com/fwlink/?linkid=89577). Вопросы о новой схеме предоставления обновлений для Microsoft Exchange можно направить через [Центр справки и поддержки Майкрософт](http://support.microsoft.com/).

**Почему это обновление устраняет несколько уязвимостей системы безопасности, о которых уже сообщалось?**  
Это обновление разработано для нескольких уязвимостей, поскольку для устранения их причин требуется выполнить изменения в связанных файлах. Вместо установки нескольких почти одинаковых обновлений пользователям необходимо установить только данное обновление.

**Производит ли данное обновление какие-либо изменения в функциях системы, связанные с безопасностью?**  
Да. Кроме изменений, перечисленных в разделе "Сведения об уязвимости" данного бюллетеня, это обновление содержит изменение для обеспечения многоуровневой защиты для Microsoft Exchange 2000 Server с пакетом обновления 3 (SP3), которое устраняет проблему с китайским набором символов при использовании Outlook Web Access (OWA). Этой проблеме подвержены только компьютеры под управлением Exchange 2000. Обратите внимание, что для установки этого обновления в Microsoft Exchange 2000 Server с пакетом обновления 3 (SP3) в качестве необходимого условия требуется установка накопительного пакета обновления для Exchange Server 2000, выпущенного после пакета обновления 3 (SP3). Перед установкой предоставленного в данном бюллетене обновления, обеспечивающего повышенную защиту, необходимо установить накопительный пакет обновления для Exchange 2000 (KB870540). Данное обновление безопасности определит, установлен ли этот накопительный пакет обновления. Если накопительный пакет обновления не установлен, пользователь будет направлен на веб-сайт загрузки. Дополнительные сведения см. в [статье 870540 базы знаний Майкрософт](http://support.microsoft.com/kb/870540). Для загрузки необходимого предварительного обновления см. [Накопительный пакет обновлений для Exchange 2000 (KB870540)](http://www.microsoft.com/downloads/details.aspx?familyid=363a57a4-8bed-4bbb-bbe4-abc11ab04611).

Это обновление также содержит изменение для обеспечения многоуровневой защиты для Microsoft Exchange 2007 и Microsoft Exchange 2010, вносящее дополнительную энтропию исходных портов в транзакции DNS, инициируемые службой SMTP.

**Нужно ли мне применять обновления и для Windows, и для Exchange?**  
Для компьютерах, на которых установлена служба Microsoft Exchange, следует применить как обновление для Exchange, так и обновление для Windows. Если служба SMTP включена, а служба Exchange не запущена, то нужно применить только обновление для Windows.

**Я использую Exchange 2007 или Exchange 2010. Почему мне предлагается обновление, если эти системы не подвержены уязвимостям, описанным в этом бюллетене?**  
Обновления для Microsoft Exchange 2007 и Microsoft Exchange 2010 содержат только изменение для обеспечения многоуровневой защиты, вносящее дополнительную энтропию исходных портов в транзакции DNS, инициируемые службой SMTP.

**Существует ли определенный порядок установки накопительного пакета обновления для серверов под управлением Exchange Server 2007?**  
См. статью TechNet [Порядок установки последнего пакета обновления или накопительного пакета обновления для Exchange 2007](http://technet.microsoft.com/en-us/library/ee221147(exchg.80).aspx).

**На моем компьютере установлена более ранняя версия программного обеспечения, описанного в этом бюллетене по безопасности. Что мне следует сделать?**  
Продукты, перечисленные в этом бюллетене, проверены на наличие уязвимости в тех или иных версиях. Жизненные циклы поддержки прочих версий программного обеспечения истекли. Уязвимость способа обработки записей MX SMTP-сервером (CVE-2010-0024)
---------------------------------------------------------------------

<span></span>
В способе, используемом компонентом SMTP Microsoft Windows для обработки особым образом созданных записей ресурсов MX (Mail Exchanger) DNS, существует уязвимость, делающая возможными атаки типа "отказ в обслуживании". Чтобы воспользоваться этой уязвимостью, злоумышленнику не нужно проходить проверку подлинности, а достаточно лишь отправить специально созданное сообщение по сети на компьютер со службой SMTP. В случае успешной атаки с использованием этой уязвимости служба SMTP перестанет отвечать на запросы до тех пор, пока она не будет перезапущена.

Чтобы просмотреть стандартную запись об этой уязвимости в списке "Common Vulnerabilities and Exposures", см. [CVE-2010-0024](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2010-0024).

#### Смягчающие факторы для уязвимости способа обработки записей MX SMTP-сервером (CVE-2010-0024)

К факторам, снижающим опасность уязвимости, относятся параметр, стандартная конфигурация или общие рекомендации, которые при использовании по умолчанию могут снизить опасность использования уязвимости. Следующие факторы могут снизить опасность использования уязвимости.

-   По умолчанию компонент SMTP в системах Windows Server 2003, Windows Server 2003 x64 Edition и Windows XP Professional x64 Edition не устанавливается.

#### Временные решения для уязвимости способа обработки записей MX SMTP-сервером (CVE-2010-0024)

Корпорацией Майкрософт не найдено временных решений для устранения этой уязвимости.

#### Часто задаваемые вопросы для уязвимости способа обработки записей MX SMTP-сервером (CVE-2010-0024)

**Какова область воздействия данной уязвимости?**  
Эта уязвимость позволяет провести атаку типа «отказ в обслуживании». Если злоумышленник воспользуется этой уязвимостью, пораженная служба SMTP перестанет отвечать на запросы до тех пор, пока особым образом созданное сообщение не будет удалено из очереди, а служба не будет перезапущена вручную. Обратите внимание, что уязвимость типа "отказ в обслуживании" не позволяет злоумышленнику выполнить код или повысить уровень своих прав, но может привести к тому, что система перестанет принимать запросы.

**В чем причина уязвимости?** Данная уязвимость вызвана тем, что компонент SMTP Microsoft Windows неверно анализирует особым образом созданную запись MX (Mail Exchanger) DNS.

**Что такое служба SMTP?**  
Служба SMTP (Simple Mail Transfer Protocol), передающая сообщения электронной почты, устанавливается как часть служб электронной почты или Internet Information Services (IIS). SMTP управляет способами транспортировки и последующей доставки электронной почты через Интернет на целевой сервер. Служба SMTP отправляет и принимает сообщения электронной почты между серверами. Дополнительные сведения см. в статье TechNet [Служба SMTP (Simple Mail Transfer Protocol)](http://technet.microsoft.com/en-us/library/cc759281(ws.10).aspx).

**Что такое запись ресурса MX?**  
Запись ресурса MX (Mail Exchange) задает почтовый сервер для доменного имени DNS. Почтовый сервер — это узел, обрабатывающий и пересылающий почту для данного доменного имени DNS. Обработка почты означает либо ее доставку адресату, либо ее передачу в другую систему транспорта почты. Пересылка почты означает ее передачу на целевой сервер, передачу с использованием SMTP (Simple Mail Transfer Protocol) на другой почтовый сервер ближе к пункту назначения или размещение в очереди на указанное время. Дополнительные сведения о записях ресурсов MX см. в статье TechNet [Типы записей ресурсов](http://technet.microsoft.com/en-us/library/cc958958.aspx).

**Что может сделать злоумышленник, который воспользуется этой уязвимостью?**  
Если злоумышленник успешно воспользуется этой уязвимостью, служба SMTP перестанет отвечать на запросы до тех пор, пока особым образом созданное сообщение не будет удалено из очереди, а служба не будет перезапущена вручную.

**Каким образом злоумышленник может воспользоваться этой уязвимостью?**  
Злоумышленник может попытаться воспользоваться этой уязвимостью, создав вредоносный DNS-сервер, который возвращает особым образом созданный ответ на запрос записи ресурсов MX.

**Какие системы в первую очередь подвергаются риску?**  
Риску из-за этой уязвимости подвержены все операционные системы, использующие собственный компонент SMTP, который поставляется как часть операционной системы.

**Как действует обновление?**  
Данное обновление для системы безопасности устраняет эту уязвимость, исправляя способ, используемый службой SMTP для анализа записей MX.

**Было ли объявлено о данной уязвимости до выпуска этого бюллетеня по безопасности?**  
Да. О данной уязвимости сообщалось в открытых источниках. В списке "Common Vulnerabilities and Exposures" данной уязвимости присвоен номер [CVE-2010-0024](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2010-0024).

**Получала ли корпорация Майкрософт к моменту выпуска этого бюллетеня по безопасности какие-либо сведения о том, что уязвимость была использована злоумышленниками?**  
Нет. Корпорация Майкрософт не получала никакой информации, указывающей на то, что до выпуска этого бюллетеня по безопасности данная уязвимость была открыто использована для организации атак на пользовательские компьютеры.

Уязвимость способа выделения памяти службой SMTP (CVE-2010-0025)
----------------------------------------------------------------

<span></span>
Из-за способа, который компонент SMTP использует для выделения памяти, в компоненте SMTP (Simple Mail Transfer Protocol) Microsoft Windows существует уязвимость, приводящая к раскрытию информации. Злоумышленник может воспользоваться данной уязвимостью, отправив на уязвимый сервер недопустимые команды, заканчивающиеся командой STARTTLS. Воспользовавшись этой уязвимостью, злоумышленник может прочесть случайные фрагменты сообщений электронной почты, хранящиеся на уязвимом сервере. Отметим, что эта уязвимость не позволяет злоумышленнику выполнять код или непосредственно повышать уровень своих пользовательских прав, но она может быть использована для выявления полезной информации, которую можно было бы использовать для дальнейших атак на уязвимую систему.

Чтобы просмотреть стандартную запись об этой уязвимости в списке "Common Vulnerabilities and Exposures", см. [CVE-2010-0025](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2010-0025).

#### Смягчающие факторы для уязвимости способа выделения памяти службой SMTP (CVE-2010-0025)

#### Временные решения для уязвимости способа выделения памяти службой SMTP (CVE-2010-0025)

Корпорацией Майкрософт не найдено временных решений для устранения этой уязвимости.

#### Часто задаваемые вопросы об уязвимости способа выделения памяти службой SMTP (CVE-2010-0025)

**Какова область воздействия данной уязвимости?**  
Эта уязвимость связана с возможностью утечки информации. Воспользовавшись этой уязвимостью, злоумышленник может прочесть случайные фрагменты сообщений электронной почты, хранящиеся на уязвимом сервере. Отметим, что эта уязвимость не позволяет злоумышленнику выполнять код или непосредственно повышать уровень своих пользовательских прав, но она может быть использована для выявления полезной информации, которую можно было бы использовать для дальнейших атак на уязвимую систему.

**В чем причина уязвимости?** Данная уязвимость вызвана тем, что компонент SMTP (Simple Mail Transfer Protocol) Microsoft Windows неправильно выделяет память при интерпретации ответов на команды SMTP.

**Что такое служба SMTP?**  
Служба SMTP (Simple Mail Transfer Protocol), передающая сообщения электронной почты, устанавливается как часть служб электронной почты или Internet Information Services (IIS). SMTP управляет способами транспортировки и последующей доставки электронной почты через Интернет на целевой сервер. Служба SMTP отправляет и принимает сообщения электронной почты между серверами. Дополнительные сведения см. в статье TechNet [Служба SMTP (Simple Mail Transfer Protocol)](http://technet.microsoft.com/en-us/library/cc759281(ws.10).aspx).

**Что такое протокол TLS?**  
Протокол подтверждения соединения TLS отвечает за проверку подлинности и обмен ключами, необходимые для установления или возобновления безопасных соединений. При установлении безопасного соединения протокол подтверждения соединения обеспечивает управление следующими процессами:

-   согласование комплекта шифров;
-   проверка подлинности сервера и (необязательно) клиента;
-   обмен сведениями о ключах сеанса.

Дополнительные сведения см. в статье [Как работает TLS/SSL](http://technet.microsoft.com/en-us/library/cc783349(ws.10).aspx) на веб-сайте Technet.

**Что такое SSL?**  
SSL (Secure Sockets Layer) — это предшественник протокола TLS (Transport Layer Security). SSL выполняет те же функции и поддерживает защищенные сетевые взаимодействия, используя технологию создания открытого и секретного ключей. Дополнительные сведения см. в статье [Как работает TLS/SSL](http://technet.microsoft.com/en-us/library/cc783349(ws.10).aspx) на веб-сайте Technet.

**Что такое пакет безопасности SChannel?**  
Пакет безопасности Secure Channel (SChannel) — это поставщик поддержки безопасности (SSP), реализующий стандартные интернет-протоколы проверки подлинности SSL и TLS. Эти компоненты используются для безопасного обмена данными при поддержке некоторых общих интернет-приложений и сетевых приложений, например веб-браузеров. Безопасный канал Schannel является частью пакета безопасности, предоставляющего службу проверки подлинности для обеспечения безопасного обмена данными между клиентом и сервером. Дополнительные сведения см. в статье MSDN [Secure Channel](http://msdn.microsoft.com/en-us/library/aa380123(vs.85).aspx) (Безопасный канал).

**Что может сделать злоумышленник, который воспользуется этой уязвимостью?**  
Воспользовавшись этой уязвимостью, злоумышленник может получить доступ к случайным фрагментам сообщений электронной почты, имеющимся на уязвимом сервере.

**Каким образом злоумышленник может воспользоваться этой уязвимостью?**  
Злоумышленник может попытаться воспользоваться этой уязвимостью, отправив набор особым образом созданных команд SMTP на уязвимый SMTP-сервер. В этом случае он сможет считывать информацию из памяти уязвимой системы.

**Какие системы в первую очередь подвергаются риску?**  
Риску из-за этой уязвимости подвержены все операционные системы, использующие собственный компонент SMTP, который поставляется как часть операционной системы.

**Как действует обновление?**  
Это обновление для системы безопасности устраняет уязвимость, исправляя способ, которым компонент SMTP выделяет память для интерпретации ответов на команды SMTP.

**Было ли объявлено о данной уязвимости до выпуска этого бюллетеня по безопасности?**  
Нет. Корпорация Майкрософт получила сведения об этой уязвимости из достоверных источников.

**Получала ли корпорация Майкрософт к моменту выпуска этого бюллетеня по безопасности какие-либо сведения о том, что уязвимость была использована злоумышленниками?**  
Нет. Корпорация Майкрософт не получала никакой информации, указывающей на то, что до выпуска этого бюллетеня по безопасности данная уязвимость была открыто использована для организации атак на пользовательские компьютеры.

**Проверка успешного применения обновления**

-   **Анализатор безопасности Microsoft Baseline Security Analyzer**

    Для проверки успешного применения обновления безопасности можно воспользоваться средством Microsoft Baseline Security Analyzer (MBSA). Дополнительные сведения см. выше в разделе **"Руководство и средства по диагностике и развертыванию"** данного бюллетеня.

-   **Проверка версий файлов**

    Эти действия зависят от установленного выпуска Microsoft Windows. В случае затруднений для выполнения процедуры обратитесь к документации по установленной версии продукта.

    1.  Нажмите кнопку **Пуск** и выберите пункт **Найти**.
    2.  На панели **Результаты поиска** в области **Помощник по поиску** выберите пункт **Файлы и папки**.
    3.  В поле **Часть имени или имя файла целиком** введите имя файла из соответствующей справочной таблицы и нажмите **Найти**.
    4.  В списке файлов щелкните правой кнопкой мыши имя файла из соответствующей таблицы сведений о файлах и выберите **Свойства**.
        **Примечание**. В зависимости от выпуска операционной системы или имеющихся на компьютере программ некоторые файлы из таблицы сведений о файлах могут быть не установлены.
    5.  На вкладке **Версия** определите версию файла, установленного на компьютере, сравнив ее с версией, приведенной в таблице сведений о файлах.
        **Примечание**. Все атрибуты, кроме версии файла, в ходе установки могут изменяться. Проверка того, что обновление было применено, путем сравнения других атрибутов файла с данными из таблицы сведений о файлах не поддерживается. Кроме того, при определенных условиях файлы в ходе установки могут быть переименованы. Если сведения о файле или версии отсутствуют, обратитесь к другим доступным методам проверки установки обновления.

-   **Проверка разделов реестра**

    Файлы, установленные этим обновлением для системы безопасности, также можно проверить в разделах реестра, перечисленных в **справочной таблице** данного раздела.

    Эти разделы реестра могут не содержать полного списка установленных файлов. Кроме того, эти разделы реестра могут быть созданы неправильно, если администратор или сборщик систем интегрируют или добавляют данное обновление для системы безопасности в исходные установочные файлы Windows.

При установке этого обновления для системы безопасности проверяется, не выполнялось ли обновление этого файла (файлов) ранее с помощью исправления Microsoft.

    Эти действия зависят от установленного выпуска Microsoft Windows. В случае затруднений для выполнения процедуры обратитесь к документации по установленной версии продукта.

Если сведения о файле или версии отсутствуют, обратитесь к другим доступным методам проверки установки обновления.

#### Сведения о развертывании
  
**Установка обновления**
  
При установке данного обновления для системы безопасности программа-установщик проверяет, не выполнялось ли ранее обновление одного или нескольких обновляемых файлов с помощью исправления Майкрософт.
  
Если один из файлов был обновлен ранее другим исправлением, установщик копирует на компьютер файлы RTMQFE, SP1QFE и SP2QFE. В противном случае будут скопированы файлы RTMGDR, SP1GDR или SP2GDR. Обновления для системы безопасности могут не содержать всех этих вариантов. Дополнительную информацию о поведении такого рода см. в [статье 824994 базы знаний Майкрософт](http://support.microsoft.com/kb/824994).
  
Дополнительные сведения об установщике см. на [веб-сайте Microsoft TechNet](http://go.microsoft.com/fwlink/?linkid=38951).
  
Подробнее о терминологии, встречающейся в этом бюллетене, например о том, что такое *исправление*, см. в [статье 824684 базы знаний Майкрософт](http://support.microsoft.com/kb/824684).

**Проверка успешного применения обновления**

-   **Анализатор безопасности Microsoft Baseline Security Analyzer**

    Для проверки успешного применения обновления безопасности можно воспользоваться средством Microsoft Baseline Security Analyzer (MBSA). Дополнительные сведения см. выше в разделе **"Руководство и средства по диагностике и развертыванию"** данного бюллетеня.

-   **Проверка версий файлов**

    Выполнение указанных ниже действий зависит от установленной версии или выпуска Microsoft Exchange. В случае затруднений для выполнения процедуры обратитесь к документации по установленной версии продукта.

    1.  Нажмите кнопку **Пуск** и выберите пункт **Найти**.
    2.  На панели **Результаты поиска** в области **Помощник по поиску** выберите пункт **Файлы и папки**.
    3.  В поле **Часть имени или имя файла целиком** введите имя файла из соответствующей справочной таблицы и нажмите **Найти**.
    4.  В списке файлов щелкните правой кнопкой мыши имя файла из соответствующей таблицы сведений о файлах и выберите **Свойства**.
        **Примечание.** В зависимости от версии операционной системы или установленных программ некоторые из файлов, перечисленных в таблице сведений о файлах, могут быть не установлены.
    5.  На вкладке **Версия** определите версию файла, установленного на компьютере, сравнив ее с версией, приведенной в таблице сведений о файлах.
        **Примечание**. Все атрибуты, кроме версии файла, в ходе установки могут изменяться. Проверка того, что обновление было применено, путем сравнения других атрибутов файла с данными из таблицы сведений о файлах не поддерживается. Кроме того, при определенных условиях файлы в ходе установки могут быть переименованы. Если сведения о файле или версии отсутствуют, обратитесь к другим доступным методам проверки установки обновления.

-   **Проверка разделов реестра**

    Файлы, установленные этим обновлением для системы безопасности, также можно проверить в разделах реестра, перечисленных в **справочной таблице** данного раздела.

    Эти действия зависят от установленной версии и выпуска системы Microsoft Windows. В случае затруднений для выполнения процедуры обратитесь к документации по установленной версии продукта.

    1.  Нажмите кнопку **Пуск** и выберите пункт **Найти**.
    2.  На панели **Результаты поиска** в области **Помощник по поиску** выберите пункт **Файлы и папки**.
    3.  В поле **Часть имени или имя файла целиком** введите имя файла из соответствующей справочной таблицы и нажмите **Найти**.
    4.  В списке файлов щелкните правой кнопкой мыши имя файла из соответствующей таблицы сведений о файлах и выберите **Свойства**.
        **Примечание.** В зависимости от версии операционной системы или установленных программ некоторые из файлов, перечисленных в таблице сведений о файлах, могут быть не установлены.
    5.  На вкладке **Версия** определите версию файла, установленного на компьютере, сравнив ее с версией, приведенной в таблице сведений о файлах.
        **Примечание**. Все атрибуты, кроме версии файла, в ходе установки могут изменяться. Проверка того, что обновление было применено, путем сравнения других атрибутов файла с данными из таблицы сведений о файлах не поддерживается. Кроме того, при определенных условиях файлы в ходе установки могут быть переименованы. Если сведения о файле или версии отсутствуют, обратитесь к другим доступным методам проверки установки обновления.

**Проверка успешного применения обновления**

-   **Анализатор безопасности Microsoft Baseline Security Analyzer**

    Для проверки успешного применения обновления безопасности можно воспользоваться средством Microsoft Baseline Security Analyzer (MBSA). Дополнительные сведения см. выше в разделе **"Руководство и средства по диагностике и развертыванию"** данного бюллетеня.

-   **Проверка версий файлов**

    Эти действия зависят от установленного выпуска Microsoft Windows. В случае затруднений для выполнения процедуры обратитесь к документации по установленной версии продукта.

    1.  Нажмите кнопку **Пуск** и затем введите имя файла обновления в поле **Начать поиск**.
    2.  Когда файл появится в меню **Программы**, щелкните имя файла правой кнопкой мыши и выберите пункт **Свойства**.
    3.  Сравните размер файла на вкладке **Общие** с данными таблиц сведений о файлах, приведенных в разделе статьи базы знаний этого бюллетеня.
    4.  Также можно открыть вкладку **Дополнительно** и сравнить версии файлов и дату их изменения с данными, приведенными в таблицах сведений о файлах раздела статьи базы знаний этого бюллетеня.
    5.  Наконец, также можно открыть вкладку **Предыдущие версии** и сравнить предыдущие версии файлов с данными о новых или обновленных версиях.

-   **Проверка разделов реестра**

    Файлы, установленные этим обновлением для системы безопасности, также можно проверить в разделах реестра, перечисленных в **справочной таблице** данного раздела.

    Эти разделы реестра могут не содержать полного списка установленных файлов. Кроме того, эти разделы реестра могут быть созданы неправильно, если администратор или сборщик систем интегрируют или добавляют данное обновление для системы безопасности в исходные установочные файлы Windows.
