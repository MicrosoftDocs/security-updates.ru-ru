Security Bulletin

Бюллетень по безопасности (Майкрософт) MS11-100 - Критическое
=============================================================

Уязвимости в .NET Framework делают возможным несанкционированное получение прав (2638420)
-----------------------------------------------------------------------------------------

Дата публикации: 29 декабря 2011 г. | Дата обновления: 10 июля 2012 г.

**Версия:** 1.6

### Общие сведения

#### Аннотация

Это обновление для системы безопасности устраняет одну опубликованную и три обнаруженных пользователями уязвимостей в Microsoft .NET Framework. Наиболее серьезная из этих уязвимостей делает возможным несанкционированное получение прав, если злоумышленник, не прошедший проверку подлинности, отправляет специально созданный веб-запрос целевому веб-сайту. Воспользовавшись этой уязвимостью, злоумышленник может выполнить любое действие в контексте существующей учетной записи на сайте ASP.NET, включая выполнение произвольных команд. Для использования этой уязвимости злоумышленник должен обладать возможностью регистрации учетной записи на сайте ASP.NET и знать имя существующего пользователя.

Это обновление для системы безопасности имеет уровень серьезности "критический" для Microsoft .NET Framework 1.1 с пакетом обновления 1 (SP1), Microsoft .NET Framework 2.0 с пакетом обновления 2 (SP2), Microsoft .NET Framework 3.5 с пакетом обновления 1 (SP1), Microsoft .NET Framework 3.5.1 и Microsoft .NET Framework 4 во всех поддерживаемых выпусках Microsoft Windows. Дополнительные сведения см. в подразделе **Подвержены и не подвержены уязвимости** далее в этом разделе.

Это обновление для системы безопасности устраняет указанные уязвимости, исправляя то, как в .NET Framework обрабатываются специально созданные запросы, а в ASP.NET Framework выполняется проверка подлинности пользователей и обрабатывается кэшированное содержимое. Дополнительные сведения об этих уязвимостях см. в подразделе ответов на часто задаваемые вопросы для каждой уязвимости, упомянутой в разделе **Сведения об уязвимостях**.

Кроме того, это обновление устраняет уязвимость, впервые описанную в [выпуске 2659883 советов по безопасности (Microsoft)](http://technet.microsoft.com/security/advisory/2659883).

**Рекомендация.** У большинства клиентов включено автоматическое обновление, поэтому нет необходимости предпринимать какие-либо действия: данное обновление для системы безопасности загрузится и установится автоматически. Клиентам, у которых не включено автоматическое обновление, необходимо проверить наличие обновлений и установить это обновление вручную. Дополнительные сведения об особых параметрах конфигурации автоматического обновления см. в [статье 294871 базы знаний Майкрософт](http://support.microsoft.com/kb/294871).

Предприятиям, а также администраторам и конечным пользователям, желающим применить данное обновление для системы безопасности вручную, корпорация Майкрософт рекомендует установить это обновление, запустив соответствующее ПО для управления обновлениями или проверив наличие обновлений на веб-сайте [Центра обновления Майкрософт](http://go.microsoft.com/fwlink/?linkid=40747).

См. также раздел **Руководство и средства по диагностике и развертыванию** этого бюллетеня.

**Известные проблемы.** В [статье 2638420 базы знаний Майкрософт](http://support.microsoft.com/kb/2638420) описаны известные на данный момент проблемы, с которыми пользователи могут столкнуться при установке этого обновления для системы безопасности. В этой статье также приводятся решения, рекомендованные для устранения этих проблем.

#### Подвержены и не подвержены уязвимости

Следующие продукты были проверены на наличие уязвимости в тех или иных версиях и выпусках. Прочие версии или выпуски не подвержены уязвимости, либо жизненные циклы их поддержки истекли. Сведения о жизненных циклах поддержки версий или выпусков используемых программных продуктов см. на веб-странице [сроков поддержки продуктов Майкрософт](http://go.microsoft.com/fwlink/?linkid=21742).

Часто задаваемые вопросы о данном обновлении безопасности
---------------------------------------------------------

<span></span>
**Почему выпущена новая редакция этого бюллетеня по безопасности** **от22мая, 2012 г.?**
Новая редакция этого бюллетеня выпущена потому, что в нее включена информация об изменении логики обнаружения для обновления KB2656352 для Microsoft .NET Framework 2.0 с пакетом обновления 2 (SP2) для устранения проблемы установки. Это изменение касается только логики обнаружения. В файлы обновления безопасности изменений не вносилось. Пользователям, уже выполнившим успешное обновление компьютеров, не требуется выполнять никаких действий.

**Связан ли этот бюллетень с бюллетенем MS12-035, Уязвимости в .NET Framework делают возможным удаленное выполнение кода (2693777)?**
Да. Обновление KB2656353, предлагаемое в этом бюллетене, также устраняет уязвимости CVE-2012-0160 и CVE-2012-0161, описанные в бюллетене [MS12-035](http://go.microsoft.com/fwlink/?linkid=246970).

**Подвержены ли этой уязвимости Microsoft .NET Framework 3.5 и Microsoft .NET Framework 4.5** **на Windows** **Developer** **Preview?**
Да. Сообщения об этой уязвимости поступили после выпуска Windows Developer Preview. Пользователям Developer Preview рекомендуется загрузить и установить это обновление на своих компьютерах. Дополнительные сведения, включая ссылки Центра загрузки Майкрософт, см. в следующих статьях базы знаний Майкрософт:

| Подвержены уязвимости                                     | Статья базы знаний Майкрософт                                                    |
|-----------------------------------------------------------|----------------------------------------------------------------------------------|
| Microsoft .NET Framework 3.5 на Windows Developer Preview | [Статья 2656360 базы знаний Майкрософт](http://support.microsoft.com/kb/2656360) |
| Microsoft .NET Framework 4.5 на Windows Developer Preview | [Статья 2656354 базы знаний Майкрософт](http://support.microsoft.com/kb/2656354) |

Обновления для системы безопасности доступны в [Центре обновления Майкрософт](http://go.microsoft.com/fwlink/?linkid=40747) и [](http://go.microsoft.com/fwlink/?linkid=21130)[Центре обновления Windows](http://go.microsoft.com/fwlink/?linkid=21130)</a>.

**Отражаются ли вносимые данным обновлением изменения системы безопасности на функциональности?**
Да. Кроме изменений, перечисленных в разделе "Сведения об уязвимости" данного бюллетеня, это обновление содержит функциональные изменения для ASP.NET Framework. Данное обновление ограничивает максимальное число ключей форм, файлов и членов JSON: не более 1000 в одном запросе. В результате данного изменения приложения ASP.NET отклоняют запросы, в которых более 1000 таких элементов. Клиенты, выполняющие запросы данных типов, отклоняются, а в их браузерах отображается сообщение об ошибке (обычно ошибка 500). Данное ограничение можно настроить для каждого приложения. См. [статью 2661403 базы знаний Майкрософт](http://support.microsoft.com/kb/2661403).

Данное обновление также изменяет формат билетов проверки подлинности с помощью форм и они перестают быть совместимыми с более старой версией билетов проверки подлинности с помощью форм. Например, после установки данного обновления в средах веб-ферм, в которых одни серверы обновлены, а другие нет, возможна ситуация, когда обновленные серверы создают билеты проверки подлинности с помощью форм, которые не совместимы с билетами, созданными необновленными серверами. См. [статью 2661404 базы знаний Майкрософт](http://support.microsoft.com/kb/2661404).

Местонахождение дополнительных сведений о файлах см. в справочных таблицах раздела **Развертывание обновления для системы безопасности**.

**Как определить, какая версия Microsoft .NET Framework установлена на моем компьютере?**
Пользователь может установить и запустить несколько версий .NET Framework на компьютере, причем установка версий допускается в любом порядке. Существует несколько способов определения того, какие версии .NET Framework установлены на компьютере. Дополнительные сведения см. в [статье 318785 базы знаний Майкрософт](http://support.microsoft.com/kb/318785).

**Чем отличаются .NET Framework 4 и клиентский профиль .NET Framework 4?**
Предлагаются два профиля для распространяемых пакетов .NET Framework версии 4: .NET Framework 4 и клиентский профиль .NET Framework 4. Клиентский профиль .NET Framework 4 является подмножеством профиля .NET Framework 4, оптимизированным для клиентских приложений. В нем содержатся функции для большинства клиентских приложений, включая функции Windows Presentation Foundation (WPF), Windows Forms, Windows Communication Foundation (WCF) и ClickOnce. Это позволяет ускорить развертывание и уменьшить размер установочных пакетов для приложений, предназначенных для клиентского профиля .NET Framework 4. Дополнительные сведения см. в статье MSDN [Клиентский профиль .NET Framework](http://msdn.microsoft.com/en-us/library/cc656912.aspx).

**Для версии Microsoft .NET Framework, установленной в моей системе, указаны два обновления. Нужно ли мне установить оба обновления?**
Да. Пользователям следует применить все обновления, предлагаемые для установленного в их системах программного обеспечения.

**Должен ли я устанавливать эти обновления безопасности в определенном порядке?**
Нет. Когда имеется несколько обновлений для одной версии .NET Framework, их можно применять в любой последовательности. Несколько обновлений для разных версий .NET Framework рекомендуется применять в порядке возрастания номеров версий, однако такая последовательность не является обязательной.

**У меня установлена среда .NET Framework 3.0 с пакетом обновления 2 (SP2); эта версия не значится в списке уязвимого ПО в этом бюллетене. Нужно ли мне устанавливать это обновление?**
В этом бюллетене описана уязвимость слоев объектов .NET Framework 2.0 и .NET Framework 4. В установщике .NET Framework 3.0 с пакетом обновления 2 (SP2) запрограммирована установка .NET Framework 2.0 с пакетом обновления 2 (SP2), поэтому при установке первого продукта устанавливается и второй указанный продукт. Поэтому пользователи, у которых установлена среда .NET Framework 3.0 с пакетом обновления 2 (SP2), должны установить обновления безопасности для .NET Framework 2.0 с пакетом обновления 2 (SP2).

**У меня установлена среда .NET Framework 3.5 с пакетом обновления 1 (SP1). Нужно ли мне устанавливать какие-нибудь дополнительные обновления?**
В этом бюллетене описана уязвимость слоев объектов .NET Framework 2.0 и .NET Framework 4. В установщике .NET Framework 3.5 с пакетом обновления 1 запрограммирована установка .NET Framework 2.0 с пакетом обновления 2 и .NET Framework 3.0 с пакетом обновления 2. Поэтому пользователи, у которых установлена среда .NET Framework 3.5 с пакетом обновления 1 (SP1), также должны установить обновления безопасности для .NET Framework 2.0 с пакетом обновления 2 (SP2).

**На моем компьютере установлена более ранняя версия программного обеспечения, описанного в этом бюллетене по безопасности. Что мне нужно сделать?**
Продукты, перечисленные в этом бюллетене, проверены на наличие уязвимости в тех или иных версиях. Жизненные циклы поддержки прочих версий программного обеспечения истекли. Коллизии в хэш-таблицах могут стать причиной уязвимости, делающей возможной атаку типа "отказ в обслуживании" (CVE-2011-3414)
-----------------------------------------------------------------------------------------------------------------------------

<span></span>
Со способом обработки в ASP.NET Framework специально созданных запросов, вызывающих коллизию хэш-функций, связано существование уязвимости, приводящей к отказу в обслуживании. -   **Временное решение на основе конфигурации**

    Приведенное ниже временное решение основано на настройке ограничения на максимальный размер запроса, который может быть принят сервером ASP.NET от клиента. Уменьшение максимального размера запроса снижает восприимчивость сервера ASP.NET к атакам типа "отказ в обслуживании".

    **Примечание.** Прежде чем изменять конфигурацию, сделайте ее резервную копию.

    Это значение конфигурации можно глобально применить ко всем веб-сайтам ASP.NET на сервере, добавив соответствующую запись в корневой файл web.config или applicationhost.config. Другой способ: данную конфигурации можно ограничить конкретным веб-сайтом или приложением, добавив ее в файл web.config такого веб-сайта или приложения.     2.  Если приложением не используется ASP.NET ViewState, добавьте указанную ниже конфигурацию в соответствующий файл конфигурации ASP.NET, чтобы ограничить максимальный размер запроса, принимаемого сервером ASP.NET от клиента, величиной 20 КБ.

        `<configuration><system.web><httpRuntime maxRequestLength="20”/></system.web></configuration>`

    Дополнительные сведения о параметре конфигурации maxRequestLength см. в документации MSDN, тема [Элемент httpRuntime](http://msdn.microsoft.com/ru-ru/library/e1f13641.aspx).

    **Примечание.** Увеличение заданных по умолчанию пределов размера строк запросов и заголовков запросов HTTP вызывает увеличение восприимчивости сервера к проблеме отказа в обслуживании, описанной в этом выпуске советов по безопасности. Дополнительные сведения о настройке этих предельных значений см. в статье [Настройки реестра Http.sys для IIS](http://support.microsoft.com/kb/820129).

    **Побочные эффекты** Клиентские запросы, размер которых превосходит значение настроенного параметра maxRequestLength, вызывают исключение ConfigurationErrorsException на стороне сервера, а клиенту возвращается состояние ошибки HTTP. Настроенные ранее предельные значения следует использовать в качестве основы и могут корректироваться для предотвращения сбоев приложения. С уменьшением значения параметра maxRequestLength снижается восприимчивость сервера к данному конкретному состоянию отказа в обслуживании. Приложения, принимающие клиентские передачи файлов, и те, которые генерируют большое состояние вида, служат примерами приложений, на которых данное временное решение негативно влияет.

    **Удаление временного решения:** Чтобы отменить временное решение, удалите разделы конфигураций, добавленные при реализации временного решения, или восстановите резервную копию конфигурации, созданную до применения временных решений.

Злоумышленник, не прошедший проверку подлинности, может отправить небольшое число специально созданных запросов ASP.NET уязвимому веб-сайту ASP.NET, вызывая состояние отказа в обслуживании.

Интернет-фишинг — это способ мошенничества, при котором компьютерные пользователи раскрывают свою личную или финансовую информацию, предоставляя ее в ответ на поддельное сообщение электронной почты или вводя ее на мошенническом веб-сайте. Обычно фишинг-мошенничество начинается с сообщения электронной почты, напоминающего официальное уведомление из доверенного источника (банка, компании по выпуску кредитных карт, известного интернет-магазина и т. п.). Однако в ASP.NET версии 2.0 поддерживается проверка подлинности на основе форм без файлов "cookie", которая приводит к передаче билета в строке запроса. Дополнительные сведения см. в указанной ниже статье MSDN [Объяснение: Проверка подлинности форм в ASP.NET 2.0](http://msdn.microsoft.com/en-us/library/ff647070.aspx).

**Если на моем веб-сайте используется веб-ферма, нужно ли обновлять все** **компьютеры** **данной веб-фермы?**

Это означает, что билеты, созданные новым способом, не могут быть считаны компьютерами, использующими старый формат, и наоборот. **Можно ли** **компьютеры** **веб-фермы обновлять по одному за раз,** **а невсе одновременно?**

#### Сведения о развертывании

**Установка обновления**

При установке этого обновления для системы безопасности проверяется, не выполнялось ли обновление этого файла (файлов) ранее с помощью исправления Microsoft.

Если один из файлов был обновлен ранее другим исправлением, установщик копирует на компьютер файлы RTMQFE, SP1QFE и SP2QFE. В противном случае будут скопированы файлы RTMGDR, SP1GDR или SP2GDR. Обновления для системы безопасности могут не содержать всех этих вариантов. **Проверка успешного применения обновления**

-   **Анализатор безопасности Microsoft Baseline Security Analyzer**

    Для проверки успешного применения обновления безопасности можно воспользоваться средством Microsoft Baseline Security Analyzer (MBSA). Дополнительные сведения см. выше в разделе **Руководство и средства по диагностике и развертыванию** данного бюллетеня.

-   **Проверка версий файлов**

    Эти действия зависят от установленного выпуска Microsoft Windows. В случае затруднений для выполнения процедуры обратитесь к документации по установленной версии продукта.

    1.  Нажмите кнопку **Пуск** и выберите пункт **Найти**.
    2.  На панели **Результаты поиска** в области **Помощник по поиску** выберите пункт **Файлы и папки**.
    3.  В поле **Часть имени или имя файла целиком** введите имя файла из соответствующей справочной таблицы и нажмите **Найти**.
    4.  В списке файлов щелкните правой кнопкой мыши имя файла из соответствующей таблицы сведений о файлах и выберите **Свойства**.
        **Примечание**. В зависимости от выпуска операционной системы или имеющихся на компьютере программ некоторые файлы из таблицы сведений о файлах могут быть не установлены.
    5.  На вкладке **Версия** определите версию файла, установленного на компьютере, сравнив ее с версией, приведенной в таблице сведений о файлах.
        **Примечание**. Все атрибуты, кроме версии файла, в ходе установки могут изменяться. Проверка того, что обновление было применено, путем сравнения других атрибутов файла с данными из таблицы сведений о файлах не поддерживается. Кроме того, в отдельных случаях файлы в ходе установки могут быть переименованы. Если сведения о файле или версии отсутствуют, обратитесь к другим доступным методам проверки установки обновления.

#### Сведения о развертывании

**Установка обновления**

При установке этого обновления для системы безопасности проверяется, не выполнялось ли обновление этого файла (файлов) ранее с помощью исправления Microsoft.

Если один из файлов был обновлен ранее другим исправлением, установщик копирует на компьютер файлы RTMQFE, SP1QFE и SP2QFE. В противном случае будут скопированы файлы RTMGDR, SP1GDR или SP2GDR. Обновления для системы безопасности могут не содержать всех этих вариантов. Дополнительную информацию о поведении такого рода см. в [статье 824994 базы знаний Майкрософт](http://support.microsoft.com/kb/824994).

Дополнительные сведения об установщике см. в [статье 832475 базы знаний Майкрософт](http://support.microsoft.com/kb/832475).

Подробнее о терминологии, встречающейся в этом бюллетене, например о том, что такое *исправление*, см. в [статье 824684 базы знаний Майкрософт](http://support.microsoft.com/kb/824684).

**Проверка успешного применения обновления**

-   **Анализатор безопасности Microsoft Baseline Security Analyzer**

    Для проверки успешного применения обновления безопасности можно воспользоваться средством Microsoft Baseline Security Analyzer (MBSA). Дополнительные сведения см. выше в разделе **Руководство и средства по диагностике и развертыванию** данного бюллетеня.

-   **Проверка версий файлов**

    Эти действия зависят от установленного выпуска Microsoft Windows. В случае затруднений для выполнения процедуры обратитесь к документации по установленной версии продукта.

    1.  Нажмите кнопку **Пуск** и выберите пункт **Найти**.
    2.  На панели **Результаты поиска** в области **Помощник по поиску** выберите пункт **Файлы и папки**.
    3.  В поле **Часть имени или имя файла целиком** введите имя файла из соответствующей справочной таблицы и нажмите **Найти**.
    4.  В списке файлов щелкните правой кнопкой мыши имя файла из соответствующей таблицы сведений о файлах и выберите **Свойства**.
        **Примечание**. В зависимости от выпуска операционной системы или имеющихся на компьютере программ некоторые файлы из таблицы сведений о файлах могут быть не установлены.
    5.  На вкладке **Версия** определите версию файла, установленного на компьютере, сравнив ее с версией, приведенной в таблице сведений о файлах.
        **Примечание**. Все атрибуты, кроме версии файла, в ходе установки могут изменяться. Проверка того, что обновление было применено, путем сравнения других атрибутов файла с данными из таблицы сведений о файлах не поддерживается. Кроме того, в отдельных случаях файлы в ходе установки могут быть переименованы. Если сведения о файле или версии отсутствуют, обратитесь к другим доступным методам проверки установки обновления.

-   **Проверка реестра**

    Файлы, установленные этим обновлением для системы безопасности, также можно проверить в разделах реестра, перечисленных в **справочной таблице** данного раздела.

    Эти разделы реестра могут не содержать полного списка установленных файлов. Кроме того, эти разделы реестра могут быть созданы неправильно, если администратор или сборщик систем интегрируют или добавляют данное обновление для системы безопасности в исходные установочные файлы Windows.

**Проверка успешного применения обновления**

-   **Анализатор безопасности Microsoft Baseline Security Analyzer**

    Для проверки успешного применения обновления безопасности можно воспользоваться средством Microsoft Baseline Security Analyzer (MBSA). Дополнительные сведения см. выше в разделе **Руководство и средства по диагностике и развертыванию** данного бюллетеня.

-   **Проверка версий файлов**

    Эти действия зависят от установленного выпуска Microsoft Windows. В случае затруднений для выполнения процедуры обратитесь к документации по установленной версии продукта.

    1.  Нажмите кнопку **Пуск** и затем введите имя файла обновления в поле **Начать поиск**.
    2.  Когда файл появится в меню **Программы**, щелкните имя файла правой кнопкой мыши и выберите пункт **Свойства**.
    3.  Сравните размер файла на вкладке **Общие** с данными таблиц сведений о файлах, приведенных в разделе статьи базы знаний этого бюллетеня.
        **Примечание**. В зависимости от выпуска операционной системы или имеющихся на компьютере программ некоторые файлы из таблицы сведений о файлах могут быть не установлены.
    4.  Также можно открыть вкладку **Дополнительно** и сравнить версии файлов и дату их изменения с данными, приведенными в таблицах сведений о файлах раздела статьи базы знаний этого бюллетеня.
        **Примечание**. Все атрибуты, кроме версии файла, в ходе установки могут изменяться. Проверка того, что обновление было применено, путем сравнения других атрибутов файла с данными из таблицы сведений о файлах не поддерживается. Кроме того, в отдельных случаях файлы в ходе установки могут быть переименованы. Если сведения о файле или версии отсутствуют, обратитесь к другим доступным методам проверки установки обновления.
    5.  Наконец, также можно открыть вкладку **Предыдущие версии** и сравнить предыдущие версии файлов с данными о новых или обновленных версиях.

**Проверка успешного применения обновления**

-   **Анализатор безопасности Microsoft Baseline Security Analyzer**

    Для проверки успешного применения обновления безопасности можно воспользоваться средством Microsoft Baseline Security Analyzer (MBSA). Дополнительные сведения см. выше в разделе **Руководство и средства по диагностике и развертыванию** данного бюллетеня.

-   **Проверка версий файлов**

    Эти действия зависят от установленного выпуска Microsoft Windows. В случае затруднений для выполнения процедуры обратитесь к документации по установленной версии продукта.

    1.  Нажмите кнопку **Пуск** и затем введите имя файла обновления в поле **Начать поиск**.
    2.  Когда файл появится в меню **Программы**, щелкните имя файла правой кнопкой мыши и выберите пункт **Свойства**.
    3.  Сравните размер файла на вкладке **Общие** с данными таблиц сведений о файлах, приведенных в разделе статьи базы знаний этого бюллетеня.
        **Примечание**. В зависимости от выпуска операционной системы или имеющихся на компьютере программ некоторые файлы из таблицы сведений о файлах могут быть не установлены.
    4.  Также можно открыть вкладку **Дополнительно** и сравнить версии файлов и дату их изменения с данными, приведенными в таблицах сведений о файлах раздела статьи базы знаний этого бюллетеня.
        **Примечание**. Все атрибуты, кроме версии файла, в ходе установки могут изменяться. Проверка того, что обновление было применено, путем сравнения других атрибутов файла с данными из таблицы сведений о файлах не поддерживается. Кроме того, в отдельных случаях файлы в ходе установки могут быть переименованы. Если сведения о файле или версии отсутствуют, обратитесь к другим доступным методам проверки установки обновления.
    5.  Наконец, также можно открыть вкладку **Предыдущие версии** и сравнить предыдущие версии файлов с данными о новых или обновленных версиях.

#### Сведения о развертывании

**Установка обновления**

При установке этого обновления для системы безопасности проверяется, не выполнялось ли обновление этого файла (файлов) ранее с помощью исправления Microsoft.

Подробнее о терминологии, встречающейся в этом бюллетене, например о том, что такое *исправление*, см. в [статье 824684 базы знаний Майкрософт](http://support.microsoft.com/kb/824684).

Обновление для системы безопасности поддерживает следующие параметры установки.

| Параметр                                       | Описание                                                                                                                         |
|------------------------------------------------|----------------------------------------------------------------------------------------------------------------------------------|
| **/?, /h, /help**                              | Вывод справки о поддерживаемых параметрах.                                                                                       |
| **/quiet**                                     | Подавление отображения сообщений о состоянии или ошибке.                                                                         |
| **/norestart**                                 | При вводе с параметром **/quiet** система не перезагрузится после установки, даже если для ее завершения требуется перезагрузка. |
